## 摘要（中文翻译与解读）

### 原始摘要
The dense, temporal nature of video presents a profound challenge for automated analysis. Despite the use of powerful Vision-Language Models, prevailing methods for video understanding are limited by the inherent disconnect between reasoning and perception: they rely on static, pre-processed information and cannot主动寻求原始视频证据as their understanding evolves. To address this, we introduce LensWalk, a flexible agentic framework that empowers a Large Language Model reasoner to control its own visual observation actively. LensWalk establishes a tight reason-plan-observe loop where the agent dynamically specifies, at each step, temporal scope and sampling density of the video it observes. Using a suite of versatile, Vision-Language Model based tools parameterized by these specifications, the agent can perform broad scans for cues, focus on specific segments for fact extraction, and stitch evidence from multiple moments for holistic verification. This design allows for progressive, on-demand evidence gathering that直接服务于agent不断演化的思维链. Without requiring any model fine-tuning, LensWalk delivers substantial, plug-and-play performance gains on multiple model配方, boosting their accuracy by over 5% on challenging long-video benchmarks like LVBench and Video-MME. Our analysis reveals that enabling an agent to control how it sees is key to unlocking more准确, robust, and可解释的视频推理.

### 中文翻译
视频的密集性和时序性给自动分析带来了巨大挑战。尽管使用了强大的视觉-语言模型，但当前的视频理解方法仍然受到推理和感知之间内在脱节的限制：它们依赖静态的、预处理的信息，无法在其理解演化时主动从视频寻求原始证据。为了解决这个问题，我们引入了LensWalk，一个灵活的智能体框架，赋予大型语言模型推理器主动控制其自身视觉观察的能力。LensWalk建立了一个紧密的推理-规划-观察循环，智能体在每一步都动态指定它所观察的视频的时间范围和采样密度。使用一套由这些规范参数化的、基于视觉-语言模型的多样化工具，智能体可以执行广泛扫描以寻找线索，聚焦特定片段以提取事实，并从多个时刻拼接证据以进行整体验证。这种设计允许渐进式、按需的证据收集，直接服务于智能体不断演化的思维链。无需任何模型微调，LensWalk在多个模型配方上提供了实质性的、即插即用的性能提升，在LVBench和Video-MME等具有挑战性的长视频基准上将准确率提升了超过5%。我们的分析表明，赋予智能体控制其观看方式的能力是解锁更准确、稳健和可解释的视频推理的关键。

## 第1节：引言

### 1.1 视频理解的核心挑战

#### 问题的本质
视频理解是计算机视觉的核心任务，面临着深刻的挑战，这源于其信息的多层次、高密度和时序展开性质。这种信息量压倒了任何分析主体的有限认知资源——无论是生物的还是人工智能的——使得统一的、穷尽的分析变得不可行且低效。

#### 人类的应对方式
强大的智能（如人类）通过**有目的的信息寻求**来应对这一挑战：
- 我们不断适应——从广泛的外围扫描转向敏锐的中央凹聚焦
- 我们间歇性地反思和验证
- 由持续演化的推理引导，这种推理既改变意图也改变观察优先级

#### 核心论点
论文的核心论点是：**这种原则是实现可靠、准确视频分析的关键：观察应该在有限容量下主动调度。**然而，在实践中实现这种以目的为导向的视频观察规划仍然是一个开放挑战。

### 1.2 现有方法的局限性

#### 单一模型设置的局限
尽管视觉-语言模型（VLMs）和测试时思考的进展显著推动了领域发展，但这一原则在当前视频理解管道中远未实现。

**典型设置**：
- 长视频在上下文窗口内被采样
- 转换为单个上下文（一次性完成）
- 这种统一策略通常会：
  - 丢弃稀疏但决定性的事件
  - 用冗余信息淹没关键内容

**改进尝试的局限**：
- 近期尝试启发式帧或token选择来保留"关键"信息
- 但这些启发式方法仍然依赖：
  - 固定的、一次性采样
  - 无法在中间假设变化时转移焦点或重新分配预算
  - 因此难以支持复杂问题的逐步解决

#### 检索-based和智能体框架的局限
许多最近的检索式和智能体框架明确尝试在推理展开时扩展信息，但仍然存在问题：

**反应式检索管道**：
- 让模型触发工具来获取预处理的工件，如：
  - ASR转录
  - OCR文本
  - 片段级字幕
- **问题**：感知仍然很大程度上绑定到：
  - 固定的、手工制作的工作流
  - 预先确定的视频视图菜单
  - 在出现模糊时决定粒度或重新查看原始视频的能力有限

**带工具流的视频智能体**：
- 让LLM控制器在预先计算的视频索引上查询
- 提供更灵活的查询
- **问题**：仍然在静态表示上操作
  - 而不是从源视频调度新鲜的观察
  - 观察和推理仍然只是松散耦合
  - 整体预算仍然规划不足

### 1.3 LensWalk的核心贡献

#### 核心思想
LensWalk是一个智能体框架，它给予LLM推理器**对在哪里以及以多密集的密度观看视频的直接控制**。

#### 架构设计
LensWalk将推理组织为**多轮推理-规划-观察循环**：

**每一轮**：
1. **推理器**反思当前问题和累积证据
2. **规划一个观察动作**：
   - 选择观察工具
   - 指定参数
   - 确定**可见上下文**：
     - 源视频的哪些时间区域要检查
     - 以什么采样级别

#### 观察工具套件
这个循环由一套互补的观察工具实例化，这些工具由视觉-语言模型驱动，将准备好的上下文转换为视觉证据：

1. **Scan Search（扫描搜索）**：
   - 支持对广泛、不连续的候选范围进行并行扫描
   - 目的：线索发现

2. **Segment Focus（片段聚焦）**：
   - 为目标间隔构建紧凑的、细节导向的视图
   - 目的：事实提取

3. **Stitched Verify（拼接验证）**：
   - 将来自多个关键片段的定制样本组装成单个上下文
   - 目的：整体检查

**每个工具都鼓励返回**：
- 与发现的关键信息交织的时间戳
- 作为后续轮次的锚点

#### 记忆机制
随着观察从原始视频逐步获取，LensWalk维护一个**轮更新的全局cast表**：
- 作为轻量级热记忆
- 跨轮次跟踪实体及其关系
- 无需重复阅读长历史

### 1.4 LensWalk的优势

#### 紧密耦合
通过将每个观察绑定到显式的、参数化的计划：
- LensWalk让智能体从其演化的推理状态直接调度：
  - 在哪里看
  - 以多密集的密度采样视频
- 随着假设的变化，跨区域和时间尺度重新分配其帧预算

#### 实证结果
在长视频理解和视频推理基准上的实证结果揭示：
- 这个框架不仅在概念上吸引人
- 而且在实践中有效

**即插即用框架**：
- LensWalk持续改进甚至强大的Reasoner-Observer组合
- 当单个模型同时作为两者进行自我调度观察时：
  - 它有效地提供了"免费午餐"
  - 产生实质性增益——例如：
    - 强大的o3模型在LVBench和VideoMME（长分割）等具有挑战性的长视频基准上准确率提高了超过11.5%和6.7%

#### 效率优势
同时，LensWalk表现出管理观察成本的**涌现能力**：
- 自适应地用最少轮次解决简单查询
- 仅对真正不确定的情况分配更多预算（如图1(d)所示）
- 导致：
  - 总token使用量远低于消耗数百万token的检索智能体
  - 仅略高于对长帧序列的单一密集传递

#### 推理追踪分析
对其推理追踪的进一步分析揭示了**多样化的、类似人类的策略**：
- 战略反思
- 渐进式缩进
- 整体验证

这些策略取代了：
- 早期智能体中常见的固定工作流
- 或退化的动作重复

这确认了**性能和效率增益源于其核心原则：由推理引导的主动观察规划**。

## 第2节：相关工作

### 2.1 视频理解中的上下文选择

#### 问题背景
多模态大型语言模型（MLLMs）的受限上下文窗口激发了一系列关于选择紧凑但信息丰富的视频上下文的工作。

#### 现有方法分类

**方法1：查询感知的关键帧或片段选择**
- 选择标准：
  - 时序和语义多样性
  - 任务相关的片段

**方法2：学习任务特定的对齐模块**
- 学习选择任务相关片段的模块

**方法3：信息压缩**
- 在帧、片段或模型层粒度上修剪冗余

#### 现有方法的局限
虽然这些策略提高了有限token预算的利用率，但它们仍然**在选择可见上下文之前一次性完成选择**，而：
- **我们的智能体**将上下文选择本身视为由其演化的假设驱动的逐步的、工具中介的过程

### 2.2 视频理解中的测试时扩展（Test-Time Scaling, TTS）

#### 视频理解TTS的进展
视频理解中测试时扩展的近期进展通过在推理时分配额外计算来提高性能。

**方向1：加强内部推理**
- 训练模型支持：
  - 长的、自我纠正的思维链
  - 可验证奖励或基于偏好的目标
- 例如：Video-R1和VideoRFT
- **局限**：仍然在固定的视觉输入上操作

**方向2：将视频模型包装在智能体工作流中**
- LLM控制如何检索或聚合预处理片段和字幕
- 基于早期管道系统：
  - 查询片段、帧和元数据
- 引入了基于推理的智能体：
  - 在长视频上调度片段检索和字幕聚合
- **局限**：智能体仅决定问什么，但对如何观察视频控制很少：
  - 时间覆盖
  - 采样密度
  - 片段组成
  - 大多由预处理管道或小部分粗略工具选项固定

#### LensWalk的对比
与这两种范式相比，我们的框架通过让智能体**明确决定如何观察视频**来扩展测试时推理：
- 选择时间范围
- 采样密度
- 片段组成
- 而不是仅在固定观察阶段分配更多计算

### 2.3 多模态工具使用智能体

#### 工具使用的核心能力
工具使用是智能体的核心能力：
- 使模型能够主动调用外部工具
- 从而将其有效上下文扩展到内在参数知识之外

#### 多模态工具使用智能体的方向

**方向1：智能体搜索**
- 智能体主动查询文本和视觉源以检索增强其内部推理上下文的证据

**方向2：智能体视觉处理**
- 为智能体配备在问题解决期间转换、组合或生成图像的工具

#### 视频理解中的工具使用
对于视频理解，最近的基于工具的方法通常通过预处理代理与视频交互：
- 将视频转换为：
  - 密集字幕或片段嵌入的数据库
  - 设计索引方案
- 然后让智能体使用基于文本的工具查询这些静态存储库

**问题**：
- 在这样的系统中：
  - 智能体可以决定问什么
  - 但对如何观察视频控制很少
  - 时间覆盖、采样密度和片段组成很大程度上由预处理管道或小部分粗略工具选项固定

#### LensWalk的突破
LensWalk通过让推理器控制观察本身，将控制从固定的预处理管道转移到动态的推理过程。

## 第3节：方法

### 3.1 LensWalk架构概览

#### 3.1.1 核心思想
尽管近期取得了巨大进展，大多数视频理解管道仍然将强大的推理器与固定的观察阶段配对：
- 视觉证据的范围和粒度不适应智能体的演化的思想

#### 3.1.2 LensWalk的设计哲学
我们转而将视频理解视为**决定如何观察**的问题，LensWalk正是设计用来让智能体动态选择：
- 在哪里看
- 在什么粒度看

详见附录A中的伪代码和具体提示。

#### 3.1.3 核心组件
LensWalk将视频理解建模为由智能体驱动的主动观察过程，该智能体智能地决定：
- 在哪里看
- 看什么
- 在每个推理步骤中按什么采样密度观看视频时间轴

**核心**：
- **LLM-based推理器（MrM_r）**：
  - 分析用户查询
  - 累积证据
  - 通过调用带有可控视频范围的observe工具来制定下一个计划

- **观察工具包（𝒪）**：
  - 利用核心VLM作为其观察器（MoM_o）
  - 根据推理器的参数直接从特定可见视频上下文提取信息

- **时间戳锚点**：
  - 用于确保多轮的一致性

- **全局主体记忆表**：
  - 作为轻量级记忆机制
  - 提供证据基础

这建立了一个迭代的推理-规划-观察循环。

#### 3.1.4 工作流程
在以下小节中，我们将依次介绍：
1. 智能体如何在多轮工具使用中调度视觉观察
2. 观察工具包的设计
3. 确保多轮一致性的证据基础机制

图2在真实视频理解实例上说明了LensWalk智能体管道。

### 3.2 推理调度的视频观察

#### 3.2.1 目标
为了灵活地获取证据，LensWalk智能体通过**基于其进行中的推理参数化每个工具调用**来调度其视觉观察。

#### 3.2.2 数学形式化
在每一步t，推理器（MrM_r）整合：
- 用户查询Q
- 视频元数据V_info（如持续时间、相机视角）
- 先前动作和观察的累积历史L_{t-1}

以产生一个完全指定的计划a_t：

a_t = M_r(Q, V_info, L_{t-1})

#### 3.2.3 计划的结构
这个计划a_t := (o_t, q_t, 𝐈_t, ρ_{o_t})精确决定下一个观察的范围和预算：

**组成部分**：
1. **o_t ∈ 𝒪**：要调用的观察工具
2. **q_t**：指导子问题
3. **𝐈_t**：范围和预算控制（例如：时间间隔(s)、帧数/FPS）
4. **ρ_{o_t}**：工具特定参数

#### 3.2.4 计划执行
执行这个计划：
- 根据𝐈_t和ρ_{o_t}采样指定帧
- 与q_t一起，将它们馈送给观察器（MoM_o）
- 观察器返回证据e_t作为新观察

#### 3.2.5 历史更新
计划-观察对被追加到历史中作为新的对话轮次：

L_t = [L_{t-1}, {a_t, e_t}]

#### 3.2.6 显式范围参数的重要性
每个动作中显式包含可控范围参数𝐈_t是关键：
- 它将历史L_t转换为带有精确时间标签的结构化文档
- 此外，这个参数化调用的历史允许智能体跟踪其自己的探索进度
  - 使其能够在后续步骤中以目标未充分探索的视频部分

### 3.3 多粒度观察工具套件

#### 3.3.1 设计原则
推理器的计划由紧凑的观察工具包执行。为支持多样化观察上下文的组合，它提供了一个**最小但富有表现力的工具集**：
- Scan Search（广泛定位）
- Segment Focus（细粒度检查）
- Stitched Verify（跨片段集成）
- Finish动作

这有效地支持了第4.4节分析中推理驱动的行为。

图2说明了它们的参数和示例。

#### 3.3.2 Scan Search（扫描搜索）

**目的**：
设计用于在指定时间间隔内进行**高效、广泛的发现**。

**参数**：
1. **指导子问题（q_t）**
2. **范围控制（𝐈_t）**
3. **工具特定设置（ρ_Scan）**：
   - 需要分区策略
   - 由切片数（n_s）或每个切片的持续时间（d_s）定义

**工作原理**：
- 工具划分给定间隔
- 从每个切片稀疏采样帧
- 在逐切片基础上查询VLM

**输出**：
- 返回每个切片的摘要或相关性判断
  - 指示目标事件的可能位置

**特性**：
- 由于每个切片被独立处理，此操作本质上是可并行化的
- 非常适合：
  - 仍然不熟悉的间隔的初始探索
  - 检测对象或事件线索的存在
  - 其并行性质允许高效定位潜在相关部分

#### 3.3.3 Segment Focus（片段聚焦）

**目的**：
设计用于在**单个、连续时间间隔上执行聚焦探测**。

**参数**：
1. **范围控制（𝐈_t）**：
   - 应用密集采样策略
2. **指导子问题（q_t）**

**工作原理**：
- 设计用于提取：
  - 细粒度细节
  - 局部动态
  - 粗略观察可能遗漏的小视觉属性

**适用场景**：
- 假设验证
- 属性读取
- 消歧相似的角色和事件

**独特优势**：
- 还允许用更密集的采样配置重新检查相同间隔
  - 澄清之前模糊的观察
  - 这种灵活性在依赖预先嵌入的视频片段的方法中不容易获得

#### 3.3.4 Stitched Verify（拼接验证）

**目的**：
设计用于观察需要**从多个非连续视频片段整合证据**的事实。

**示例场景**：
- 前后对比
- 跟踪对象转移
- 验证因果叙事

**工具特定设置（ρ_Stitch）**：
- 接受段列表S = [(s_i, e_i, r_i), ...]
  - 其中每个条目通过以下方式定义片段：
    - 开始时间（s_i）
    - 结束时间（e_i）
    - 片段的特定采样率（r_i）

**工作原理**：
- 操作符将来自这些不同间隔的帧整合到单个、连贯的批次中供VLM使用

**独特能力**：
- 唯一支持非对称采样：
  - 为动作密集的片段分配更高帧率
  - 同时为过渡片段使用较低速率
- 这使智能体能够在单步中验证复杂事件链
  - 在统一视觉上下文上推理
  - 而不是仅在语言域中调和孤立的观察

#### 3.3.5 工具包的设计哲学
这三个工具的组合提供了完整的观察能力：
1. **广泛发现**（Scan Search）
2. **深度检查**（Segment Focus）
3. **跨时间推理**（Stitched Verify）

它们共同支持智能体的多样化策略，从快速扫描到深度验证。

### 3.4 长视界一致性的证据基础

#### 3.4.1 问题的来源
因为LensWalk在不同范围内运行多个观察轮次，它必须调和它们之间潜在模糊的证据。

#### 3.4.2 两个轻量级基础组件
因此，我们引入了两个轻量级基础组件：
1. **时间戳锚点**
2. **主体记忆表**

这些组件将观察到精确时间范围和全局一致的实体引用。

#### 3.4.3 工具观察中的时间戳锚点

**目的**：
在每个观察期间（如图2），详细的时间戳锚点在指定粒度内被插入到观察器的可见上下文（帧序列）中。

**实现**：
- 这些锚点作为文本直接插入在工具的可见帧内
- 提示观察器用显式时间引用（例如"at 01:15-01:40"）来基础其答案

**优势**：
- 为推理器提供细粒度时间线索
- 使其能够：
  - 在后续规划周期中高效定位证据源
  - 调度对模糊范围的重新观察
  - 修剪不相关部分以进行更精确的分析

#### 3.4.4 用于实体一致性的主体记忆表

**目的**：
在每个观察之后（如图2），一个全局主体记忆表Sub_t由LLM从新工具观察中更新的实体信息更新。

**维护方式**：
- 在主推理历史L_t之外维护
- 记录持久实体——如人或对象
  - 以及总结属性
  - 和看到的时间戳

**使用方式**：
- 在每个推理轮次之前，它被插入到上下文中并覆盖先前版本

**双重功能**：
1. **高效的实体引用**：
   - 允许推理器使用规范标识符
   - 用于高效、一致的实体引用
   - 消除了在每次推理周期中从工具历史重复识别实体的需要

2. **结构化知识库**：
   - 提供一个干净、结构化的知识库
   - 推理器咨询该库以战略性规划其后续观察

**意义**：
- 这个机制管理累积的实体信息
- 以维护跨轮次引用稳定性
- 避免来自冗长描述的冗余上下文token
- 确保对主体的理解在它渐进探索视频时保持一致

#### 3.4.5 与现有方法的对比
在智能体视频理解中维护跨多个推理轮次的稳定、可靠的记忆是具有挑战性的：
- 先前的系统经常求助于：
  - 重预扫描
  - 摘要
  - 或复杂的记账方案

这些方法产生了相当大的计算和工程开销。

#### 3.4.6 LensWalk的优势
我们的证据基础改为：
- 维护一个紧凑、结构化的状态
- 通过将证据绑定到历史中的时间范围和实体
- 允许智能体通过重新观察重新访问模糊区域（如图2所示）

**自我纠正能力**：
- 虽然观察器或记忆更新错误仍可能引入不一致性
- 但这种基础和重新观察的轻量级组合产生了：
  - 可扩展的
  - 自纠正的管道
  - 它适应推理-规划-观察循环
  - 并得到第4.3节消融研究的支持

### 4.4 工具调用行为分析

#### 4.4.1 分析动机
我们的设计通过允许智能体控制其观察范围，使其能够基于累积经验设计多样化的观察路径。

为了更好地分析这一点，我们检查智能体的工具调用，并通过考虑以下因素识别不同的行为模式：
1. 观察查询
2. 选定的具体工具
3. 每步的观察范围

#### 4.4.2 六种主要行为模式
我们将这些模式归类为六种主要模式（见图4，详见附录C的细节和示例）：

**模式1：直接询问（Direct Inquiry）**
- 在单个、自信步骤中解决任务，无需先前探索
- 典型用于高层级或时间锚定的问题

**模式2：渐进式缩进（Progressive Zoom-in）**
- 从广泛扫描开始以定位候选时刻
- 然后放大到有前景的片段以进行更详细检查

**模式3：范围划分（Scope Partitioning）**
- 将时间线划分为不同范围
- 独立分析它们
- 仅在推理级别整合结果洞察以形成结论

**模式4：战略反思（Strategic Reflection）**
- 在聚焦搜索遇到死胡同后暂停
- 重新评估方法
- 在替代区域或假设上启动新的、更广泛的扫描

**模式5：集成验证（Integrative Verify）**
- 通过在单个拼接观察中联合检查多个关键时刻来验证复杂假设
  - 实现结论的整体确认

**模式6：静态重复（Static Repetition）**
- 陷入以几乎相同的观察在相同区域上的无生产循环
  - 显示很少的进展或策略变化

#### 4.4.3 关键发现

分析这些模式揭示了几个关键发现：

**1. 难度感知的成本分配**：
- 智能体直观地将更多观察努力分配给具有挑战性的问题
- 准确率较低的困难问题与更长、更昂贵的观察追踪相关
- 而简单案例通常用更少帧甚至单步查询高效解决
- 这表明LensWalk根据每个查询的演化不确定性自适应调整其观察深度

**2. 涌现的反思策略替代低效循环**：
- 通过其观察计划中的多样化选择赋能
  - 智能体很大程度上避免了先前工作中报告的"静态重复"行为
- 当面临不确定性时，它反而调用反思模式：
  - 战略反思
  - 集成验证
  - 以扩展和整合证据
- 这些恢复机制产生比罕见循环失败更稳定和更高的准确率

**3. 观察器质量塑造策略和效率**：
- 对于固定推理器，观察器的能力是关键的性能驱动因素
- 强观察器（例如o3）：
  - 提供清晰、决定性的证据
  - 直接解决查询的意图
  - 使频繁、低成本的直接询问成为可能
- 较弱观察器（例如Qwen2.5-VL-72B）：
  - 引入模糊性
  - 推动智能体进入更复杂和昂贵的观察计划
  - 例如退回到更广泛扫描或重复验证
- 这一差距强调了高质量感知对成本有效推理的重要性

## 与Spatial AGI的关联性分析

### 6.1 空间智能的核心需求

Spatial AGI（通用空间智能）需要智能体能够：
1. **理解空间环境**：感知三维空间、物体位置和关系
2. **主动探索**：自主决定观察的位置和方式
3. **时序推理**：理解事件的时间顺序和因果关系
4. **资源高效**：在有限的计算和存储资源下操作
5. **可解释性**：能够解释其推理过程

### 6.2 LensWalk对Spatial AGI的启发

#### 6.2.1 主动观察调度

**核心概念**：
LensWalk的核心创新是让智能体**主动控制观察**：
- 在哪里看
- 以什么密度看
- 何时调整焦点

**对Spatial AGI的意义**：
这对于空间智能体至关重要：
- 机器人在物理环境中需要决定：
  - 哪个传感器角度最有用
  - 什么时候改变视角
  - 以什么分辨率观察场景
- AR/VR系统需要：
  - 根据用户意图动态调整渲染细节
  - 智能选择虚拟摄像头的位置和方向

**实现路径**：
- 将LensWalk的reason-plan-observe循环扩展到3D空间
- 参数化：
  - 观察角度（方位角、俯仰角）
  - 观察距离（拉近/拉远）
  - 观察频率（实时vs.采样）

#### 6.2.2 多粒度观察工具

**LensWalk的工具设计**：
- Scan Search：广泛发现
- Segment Focus：详细检查
- Stitched Verify：跨时空验证

**对Spatial AGI的扩展**：

1. **空间扫描（Space Scan）**：
   - 类似于Scan Search
   - 但在3D空间中操作
   - 应用场景：
     - 机器人对环境的粗略地图构建
     - AR系统对周围物体的快速定位
   - 实现：
     - Lidar扫描
     - 多角度相机拍摄
     - 分区探索策略

2. **焦点感知（Focus Perception）**：
   - 类似于Segment Focus
   - 对空间中的特定区域密集观察
   - 应用场景：
     - 阅读空间中的标签
     - 识别物体的细粒度属性
     - 精确距离测量
   - 实现：
     - 相机缩放
     - 高分辨率局部扫描
     - 主动照明调整

3. **空间拼接（Space Stitch）**：
   - 类似于Stitched Verify
   - 但在3D空间中组合观察
   - 应用场景：
     - 验证物体移动轨迹
     - 检查空间一致性
     - 构建完整3D模型
   - 实现：
     - 多视角点云融合
     - 时空数据对齐
     - 跨视角一致性检查

#### 6.2.3 时间戳锚点与空间基础

**LensWalk的时间戳锚点**：
- 在每个观察中插入时间引用
- 提供细粒度时间线索
- 帮助定位证据源

**对Spatial AGI的空间锚点**：
扩展到空间维度：

1. **空间坐标锚点**：
   - 在每个观察中嵌入3D坐标
   - 使用框架：
     - 世界坐标系（GPS + 高度）
     - 局部坐标系（机器人中心）
     - 相对坐标系（物体间距离）

2. **方向锚点**：
   - 记录观察方向
   - 使用框架：
     - 欧拉角（roll, pitch, yaw）
     - 四元数（避免万向锁）
     - 视向量

3. **尺度锚点**：
   - 记录观察尺度
   - 使用框架：
     - 视场角（FOV）
     - 分辨率（像素/米）
     - 深度范围

**Spatial AGI中的价值**：
- 让智能体：
  - 精确定位观察
  - 调度重新观察模糊区域
  - 构建一致的空间地图
  - 跨时间跟踪物体

#### 6.2.4 主体记忆表与空间实体管理

**LensWalk的主体记忆表**：
- 维护实体的持久记录
- 记录属性和观察时间戳
- 允许一致引用

**对Spatial AGI的空间实体表**：
扩展到3D空间中的实体管理：

1. **实体类型**：
   - 静态物体：桌子、椅子、墙壁
   - 动态物体：人、机器人、宠物
   - 地标：门、窗户、路标
   - 抽象位置："厨房"、"入口"

2. **属性记录**：
   - 几何：位置、大小、方向、形状
   - 视觉：颜色、纹理、材质
   - 语义：类别、功能、所有权
   - 时空：创建时间、最后看到时间、观察频率

3. **关系跟踪**：
   - 空间关系："on"、"in"、"near"、"above"
   - 时序关系："moved from"、"disappeared at"
   - 因果关系："knocked over"、"picked up"

4. **不确定性管理**：
   - 置信度：对实体存在和属性的置信水平
   - 位置不确定性：位置误差范围
   - 观察计数：观察次数（用于验证）

**Spatial AGI中的应用**：
- 机器人导航：
  - 记录障碍物位置
  - 跟踪可导航区域
  - 记忆安全vs.危险区域
- AR交互：
  - 维护虚拟物体与现实物体的注册
  - 跟踪用户交互历史
  - 支持多用户协作
- 空间推理：
  - 推断未观察到的物体位置
  - 预测物体未来移动
  - 检测环境变化

#### 6.2.5 难度感知的资源分配

**LensWalk的发现**：
- 智能体根据问题难度自适应分配观察预算
- 简单查询用少资源解决
- 困难查询获得更多资源

**对Spatial AGI的难度感知调度**：
扩展到空间探索：

1. **空间不确定性度量**：
   - 新探索区域：高不确定性→密集探索
   - 已熟悉区域：低不确定性→稀疏采样
   - 动态环境：持续不确定性→周期性重新扫描

2. **任务难度评估**：
   - 简单任务："找到红色杯子"→单次扫描
   - 复杂任务："追踪穿蓝色衣服的人5分钟"→密集连续观察
   - 抽象任务："整理客厅"→多阶段规划+验证

3. **资源优化策略**：
   - 电池有限机器人：
     - 优先关键任务
     - 降低非关键区域采样率
   - 实时AR系统：
     - 根据用户关注动态调整质量
     - 背景vs.前景分配不同资源

#### 6.2.6 涌现的认知策略

**LensWalk发现的策略**：
- 渐进式缩进
- 战略反思
- 集成验证

**对Spatial AGI的认知策略**：

1. **渐进式空间缩进（Progressive Space Zoom-in）**：
   - 从粗略地图开始
   - 缩小到感兴趣区域
   - 最终聚焦到细节
   - 示例：
     - 机器人：先构建房间布局，再找到桌子，最后定位杯子
     - AR：先识别建筑，再定位楼层，最后找到房间

2. **战略空间反思（Strategic Space Reflection）**：
   - 当假设失败时重新评估
   - 尝试替代策略
   - 示例：
     - 机器人：找不到物体→假设位置错误→回溯重新扫描
     - AR：识别失败→调整光照或角度

3. **集成空间验证（Integrative Space Verify）**：
   - 从多个视角验证假设
   - 检查跨视角一致性
   - 示例：
     - 机器人：从不同角度确认物体身份
     - AR：跨用户验证虚拟对象位置

### 6.3 LensWalk在Spatial AGI中的具体应用场景

#### 6.3.1 机器人视觉导航

**场景描述**：
机器人在未知室内环境中导航，需要找到并操作特定物体。

**LensWalk启用的能力**：

1. **初始扫描（Scan Search类比）**：
   - 机器人执行360度全景扫描
   - 识别大致区域布局
   - 标记潜在物体位置

2. **聚焦观察（Segment Focus类比）**：
   - 接近感兴趣物体
   - 缩小观察范围
   - 执行细粒度属性识别（颜色、大小、标签）

3. **验证确认（Stitched Verify类比）**：
   - 从多个角度观察物体
   - 确认身份和可操作性
   - 检查周围环境（是否有障碍物）

4. **记忆更新（主体表类比）**：
   - 记录物体位置和属性
   - 更新环境地图
   - 用于后续任务和路径规划

**优势**：
- 自适应探索：简单环境快速完成，复杂环境深入探索
- 资源高效：只在需要时密集观察
- 鲁棒性：多视角验证避免误识别

#### 6.3.2 AR/VR场景理解

**场景描述**：
AR应用需要理解用户当前空间并提供相关信息或叠加内容。

**LensWalk启用的能力**：

1. **用户意图理解**：
   - 从用户语音或手势提取查询
   - 例如："显示这附近餐厅的评分"

2. **自适应观察规划**：
   - 根据用户兴趣区域调整摄像头方向
   - 动态调整渲染质量（前景高，背景低）

3. **上下文记忆管理**：
   - 维护空间实体表
   - 跟踪已识别和未识别物体
   - 记住用户偏好

4. **渐进式信息呈现**：
   - 初始：显示概览信息
   - 缩进：用户关注时显示更多细节
   - 验证：多角度确认信息准确性

**优势**：
- 性能：只在需要时渲染高细节
- 用户体验：渐进式信息加载
- 空间一致性：跨视角保持实体识别一致

#### 6.3.3 多传感器融合

**场景描述**：
智能体结合多个传感器（RGB摄像头、深度传感器、Lidar、IMU）理解环境。

**LensWalk启用的能力**：

1. **传感器选择（工具选择类比）**：
   - 根据任务需要选择合适传感器
   - 例如：
     - 颜色识别→RGB摄像头
     - 距离测量→深度传感器
     - 环境扫描→Lidar

2. **粒度调整（参数化观察类比）**：
   - 根据距离和重要性调整采样率
   - 近处高精度，远处低精度

3. **跨传感器验证（Stitched Verify类比）**：
   - 用不同传感器验证同一假设
   - 例如：RGB检测到物体，用深度确认位置

4. **时间戳同步**：
   - 所有传感器数据添加精确时间戳
   - 用于传感器间对齐和融合

**优势**：
- 鲁棒性：多传感器交叉验证
- 效率：只在需要时激活高功耗传感器
- 精度：根据任务动态调整传感器配置

#### 6.3.4 时序事件理解

**场景描述**：
理解长时间跨度的事件（如视频监控、交通分析、用户行为追踪）。

**LensWalk启用的能力**：

1. **粗粒度时间扫描**：

   - 识别关键时间点（事件开始、变化）

2. **细粒度事件聚焦**：
   - 对关键时间点密集分析
   - 提取细节和因果关系

3. **跨时间段验证**：
   - 检查事件一致性
   - 验证假设（如移动轨迹）

4. **实体和关系跟踪**：
   - 维护长时间实体表
   - 记录实体属性变化
   - 追踪实体间关系

**优势**：
- 效率：快速浏览长视频，聚焦关键部分
- 准确性：多角度验证事件
- 可解释性：清晰的时间戳和证据基础

## 关键创新点总结

### 7.1 核心创新

#### 7.1.1 推理调度的主动观察
- **创新点**：首次让LLM推理器动态控制视频观察的粒度和范围
- **突破**：将观察从固定的预处理阶段变为推理过程的一部分
- **意义**：实现了真正的推理-感知紧密耦合

#### 7.1.2 参数化工具系统
- **创新点**：设计了一套可参数化的观察工具（Scan Search、Segment Focus、Stitched Verify）
- **突破**：通过工具参数精确控制观察的时空范围和密度
- **意义**：提供了灵活的观察能力，支持多样化策略

#### 7.1.3 轻量级记忆机制
- **创新点**：时间戳锚点+主体记忆表，无需复杂的预扫描或摘要
- **突破**：在保持性能的同时极大降低了计算开销
- **意义**：使框架即插即用，适合实际部署

### 7.2 技术创新

#### 7.2.1 多轮reason-plan-observe循环
- 不同于单次通过的方法
- 每轮都可以调整观察策略
- 支持复杂的多步推理

#### 7.2.2 难度感知的预算分配
- 智能体根据查询难度自适应分配资源
- 简单查询快速完成，复杂查询深入探索
- 实现了效率与准确性的平衡

#### 7.2.3 涌现的认知策略
- 发现了类似人类的策略：渐进式缩进、战略反思、集成验证
- 这些策略是涌现的，非硬编码的
- 证明了框架的智能性

### 7.3 实证创新

#### 7.3.1 即插即用性能提升
- 无需任何微调
- 在多个模型和基准上一致提升性能

- 总token使用远低于检索智能体

- 自适应资源分配避免浪费

#### 7.3.3 消融验证
- 系统验证了每个组件的贡献
- 证明了工具和记忆机制的互补性
- 展示了框架设计的合理性

## 局限性与未来工作

### 8.1 当前局限性

#### 8.1.1 对强推理器的依赖
**问题**：
- 开源推理器（如Qwen3-235B-A22B）在增强强观察器时效果有限
- 这暗示框架的性能高度依赖推理器的认知能力

**影响**：
- 限制了在资源受限环境中的部署
- 可能无法在低端设备上实现良好性能

#### 8.1.2 观察器质量的重要性
**问题**：
- 弱观察器（如Qwen2.5-VL-72B）导致更复杂的观察计划
- 增加了计算成本和延迟

**影响**：
- 需要高质量的VLM以获得最佳性能
- 可能限制在仅开源模型上的应用

#### 8.1.3 计算开销
**问题**：

## 个人思考与见解

### 10.1 最有趣的发现

#### 10.1.1 涌现的认知策略
最让我印象深刻的是，LensWalk展现了类似人类的认知策略，而这些策略并非显式编程：

1. **渐进式缩进**：
   - 就像人类先浏览目录再深入阅读
   - 智能体自然地从广泛扫描到细节聚焦

2. **战略反思**：
   - 当假设失败时，智能体会重新思考
   - 而不是重复同样的错误
   - 这体现了真正的智能

3. **集成验证**：
   - 从多个角度验证复杂假设
   - 就像人类从不同证据来源确认结论

**意义**：
这证明了简单的架构可以涌现复杂智能，只要设计原则正确。Reason-plan-observe循环可能是智能的通用原理，不仅适用于视频理解，也适用于广泛的任务。

#### 10.1.2 难度感知的资源分配
另一个有趣的发现是智能体的"自知之明"：

- 它知道什么时候该深入探索（困难问题）
- 也知道什么时候可以快速解决（简单问题）
- 这种自适应能力是真正智能的标志

**意义**：
这与人类的认知一致——我们不会在简单问题上浪费精力，但在困难问题上坚持不懈。让AI拥有这种"元认知"能力是一个重要进步。

#### 10.1.3 即插即用的成功
在AI研究中，即插即用的方法很少，尤其是需要多轮交互的系统。LensWalk在多个模型配置上的一致成功令人惊讶：

- o3作为推理器和观察器：+11.5%
- GPT-5：+7.1%
- Qwen2.5-VL-72B：+3.6%（用开源推理器）

**意义**：
这证明了框架设计的普适性，而非针对特定模型的"魔法参数"。这样的鲁棒性是实际应用的关键。

### 10.2 最意外的结果

#### 10.2.1 开源推理器的局限
最意外的结果是开源推理器（Qwen3-235B-A22B）在增强强观察器时的表现：

- 增强Qwen2.5-VL-7B：+4.3%（有效）
- 增强GPT-4.1：+0.1%（几乎无效）
- 增强Qwen2.5-VL-72B：-0.6%（甚至降低）

**为什么意外？**
我原以为推理器会普遍提升性能，至少不降低。但结果表明：

1. **推理器的认知能力有阈值**
   - 当观察器足够强时，弱推理器的"指导"反而成为限制
   - 观察器自身的推理能力更优

2. **推理质量 vs. 观察质量**
   - 在某些场景下，提升观察质量比提升推理质量更关键
   - 这与直觉相反

**启示**：
Spatial AGI系统需要平衡推理和感知能力。一味追求强推理器可能适得其反，需要根据观察器能力调整。

#### 10.2.2 Stitched Verify的高价值
我原以为Scan Search会是最大贡献（因为它用于初始探索），但消融结果显示：

- Scan Search移除：-4.6%（最大下降）
- **Stitched Verify移除：-3.2%（第二大下降）**
- Segment Focus移除：-1.9%

Stitched Verify的高价值令人意外，因为：
- 它只在特定场景使用（跨片段验证）
- 但却如此重要

**原因分析**：
- 复杂问题往往需要跨时空整合信息
- 单点观察无法建立完整理解
- Stitched Verify提供了这种整合能力

**启示**：
Spatial AGI需要多视角融合能力。从单一观察点无法建立可靠的空间理解。

### 10.3 最有价值的启发

#### 10.3.1 观察是智能的一部分
传统观点认为：
- 智能 = 推理能力（大脑）
- 观察是外围的（眼睛、相机）

LensWalk证明：
- **智能 = 如何观察 + 如何推理**
- 两者同样重要，且应紧密耦合

**对Spatial AGI的启示**：
在设计空间智能体时，不应只关注推理算法，必须同样重视：
- 如何选择传感器角度
- 如何调整采样密度
- 何时重新观察

#### 10.3.2 参数化控制是关键
LensWalk的核心设计原则是参数化观察：
- 时间范围
- 采样密度
- 工具类型

这种简单设计却带来了强大能力：
- 灵活性：支持多样化策略
- 可解释性：清晰的观察计划
- 控制性：精确的资源分配

**对Spatial AGI的启示**：
空间观察也应参数化：
- 观察角度（方位、俯仰、翻滚）
- 观察距离（近、中、远）
- 观察范围（视场角）
- 观察分辨率

#### 10.3.3 轻量级记忆足够
传统方法使用复杂的记忆机制：
- 向量数据库
- 检索系统
- 大规模预扫描

LensWalk只用：
- 简单的主体记忆表
- 时间戳锚点

却取得了同等或更好的效果。

**启示**：
简单往往是最好的。复杂的记忆系统可能带来：
- 计算开销
- 工程复杂性
- 引入bug的机会

在Spatial AGI中，应从简单的记忆机制开始，只在必要时增加复杂度。

#### 10.3.4 拥现智能 vs. 显式编程
LensWalk的工具调用策略是涌现的，非硬编码：

- 它"学会"了战略反思
- 它"学会"了渐进式缩进
- 它"学会"了集成验证

这与传统方法形成对比：
- 传统：设计复杂的状态机和规则
- LensWalk：提供工具和奖励，让策略自然涌现

**启示**：
Spatial AGI应支持涌现，而非预设。提供丰富的观察和动作能力，让智能体自然发展出智能策略。

### 10.4 与其他工作的对比

#### 10.4.1 vs. 检索增强（RAG）
传统RAG：
- 预处理：提前索引视频
- 查询：从索引检索

LensWalk：

## 附录

### A. 核心问题深度分析

#### Q1: 核心算法原理是什么？

**核心算法**：
LensWalk的核心算法是一个**多轮推理-规划-观察循环**：

**算法流程**：
1. **初始化**：
   - 接收用户查询Q
   - 获取视频元数据V_info
   - 初始化空历史L_0

2. **循环**（对于t = 1 到最大轮次）：
   a. **推理**：
      - 推理器M_r处理：(Q, V_info, L_{t-1})
      - 生成计划a_t = (o_t, q_t, 𝐈_t, ρ_{o_t})

   b. **终止检查**：
      - 如果o_t == Finish，返回答案

   c. **观察**：
      - 根据𝐈_t和ρ_{o_t}采样视频帧
      - 将帧+q_t馈送给观察器M_o
      - 获取证据e_t

   d. **基础**：
      - 在e_t中插入时间戳锚点
      - 更新主体记忆表Sub_t

   e. **更新历史**：
      - L_t = [L_{t-1}, {a_t, e_t}]

**算法特点**：
- 迭代式：多轮优化而非一次性决策
- 自适应：每轮根据前几轮结果调整策略
- 参数化：显式控制观察参数

**关键技术组件**：

1. **推理器（M_r）**：
   - 大型语言模型（如o3、GPT-5）
   - 负责决策和规划
   - 输入：查询、视频信息、观察历史
   - 输出：观察计划

2. **观察器（M_o）**：
   - 视觉-语言模型（如GPT-4.1、Qwen2.5-VL）
   - 负责解码视觉信息
   - 输入：视频帧 + 子问题
   - 输出：视觉证据

3. **观察工具包（𝒪）**：
   - Scan Search：广泛探索
   - Segment Focus：详细检查
   - Stitched Verify：跨片段验证
   - 每个工具通过参数控制观察范围和密度

4. **记忆机制**：
   - 时间戳锚点：细粒度时间对齐
   - 主体记忆表：实体跟踪和引用

#### Q2: 与Spatial AGI的关系？

**空间理解的层次**：

Spatial AGI需要理解空间在多个层次：

1. **几何空间**：
   - 3D坐标、距离、方向
   - 物体的位置和大小
   - 空间的拓扑结构

2. **语义空间**：
   - 物体的类别和功能
   - 空间的用途和含义
   - 物体间的关系

3. **时间空间**：
   - 事件的时序
   - 因果关系
   - 动态变化

**LensWalk的空间映射**：

| 视频维度 | 空间维度 | 映射关系 |
|---------|----------|---------|
| 时间轴 | 3D空间 | 都需要参数化观察 |
| 帧序列 | 点云/体素 | 都需要多粒度采样 |
| 时间戳锚点 | 空间坐标锚点 | 都需要证据基础 |
| 主体记忆 | 空间实体记忆 | 都需要实体跟踪 |
| 难度感知 | 空间不确定性感知 | 都需要自适应探索 |

**具体启发**：

1. **参数化观察控制**：
   - LensWalk参数：时间范围、采样密度
   - 空间扩展：观察角度、观察距离、观察分辨率

2. **多粒度探索**：
   - Scan Search → 空间扫描（粗略地图）
   - Segment Focus → 空间聚焦（细节检查）
   - Stitched Verify → 多视角验证（一致性检查）

3. **记忆和基础**：
   - 时间戳锚点 → 空间坐标锚点
   - 主体记忆表 → 空间实体表
   - 都支持精确引用和回溯

4. **自适应资源分配**：
   - 视频难度 → 空间不确定性
   - 困难区域密集探索 → 高不确定性区域密集采样
   - 简单区域快速通过 → 低不确定性区域稀疏采样

**应用场景**：

#### B.1 LensWalk主循环

```python
def LensWalk(Q, V_info, M_r, M_o, max_turns=20):
    """
    主推理循环

    Args:
        Q: 用户查询
        V_info: 视频元信息（时长、FPS等）
        M_r: 推理器（LLM）
        M_o: 观察器（VLM）
        max_turns: 最大轮次

    Returns:
        answer: 最终答案
        history: 完整观察历史
    """
    L = []  # 观察历史
    Sub_t = {}  # 主体记忆表

    for t in range(1, max_turns + 1):
        # 1. 推理：生成计划
        a_t = M_r.generate_plan(
            query=Q,
            video_info=V_info,
            history=L,
            memory=Sub_t
        )

        # 2. 检查终止
        if a_t.tool == 'Finish':
            return a_t.answer, L

        # 3. 观察：执行计划
        frames = sample_video(
            video_path=V_info.path,
            scope=a_t.scope,
            parameters=a_t.params
        )

        # 4. 感知：VLM解码视觉信息
        e_t = M_o.observe(
            frames=frames,
            question=a_t.sub_question
        )

        # 5. 基础：添加时间戳锚点
        e_t = add_timestamp_anchors(e_t, a_t.scope)

        # 6. 更新记忆：主体表
        Sub_t = update_entity_table(
            old_table=Sub_t,
            new_observation=e_t,
            scope=a_t.scope
        )

        # 7. 记录历史
        L.append({
            'turn': t,
            'action': a_t,
            'observation': e_t,
            'memory': copy(Sub_t)
        })

    return "Cannot determine answer", L
```

def scan_search(video, question, scope, partitions, M_o):
    """
    广泛扫描工具

    Args:
        video: 视频对象
        question: 指导子问题
        scope: 时间范围 (start, end)
        partitions: 分区参数（数量或时长）
        M_o: 观察器

    Returns:
        results: 每个分区的扫描结果
    """
    # 1. 划分时间范围
    if 'num_slices' in partitions:
        slices = partition_by_count(scope, partitions['num_slices'])
    else:
        slices = partition_by_duration(scope, partitions['duration'])

    # 2. 并行采样每个切片
    all_results = []
    for i, slice_range in enumerate(slices):
        frames = sample_video(video, slice_range, density='sparse')
        result = M_o.observe(
            frames=frames,
            question=question,
            context=f"Slice {i+1}/{len(slices)}"
        )
        all_results.append({
            'slice_range': slice_range,
            'summary': result.summary,
            'relevance': result.relevance_score,
            'timestamps': result.key_moments
        })

    return all_results
```

def segment_focus(video, question, scope, focus_params, M_o):
    """
    片段聚焦工具

    Args:
        video: 视频对象
        question: 指导子问题
        scope: 时间范围 (start, end)
        focus_params: 聚焦参数
        M_o: 观察器

    Returns:
        detail: 详细观察结果
    """
    # 1. 密集采样
    frames = sample_video(
        video,
        scope,
        density=focus_params.get('density', 'high'),
        fps=focus_params.get('fps', 10)
    )

    # 2. 细粒度感知
    detail = M_o.observe(
        frames=frames,
        question=question,
        mode='detailed'  # 要求细粒度分析
    )

    return {
        'range': scope,
        'key_facts': detail.facts,
        'visual_attributes': detail.attributes,
        'timestamps': detail.timestamps,
        'confidence': detail.confidence
    }

def stitched_verify(video, question, segments, M_o):
    """
    拼接验证工具

    Args:
        video: 视频对象
        question: 指导子问题
        segments: 片段列表 [(start, end, rate), ...]
        M_o: 观察器

    Returns:
        verification: 验证结果
    """
    # 1. 按不同采样率采样每个片段
    all_frames = []
    segment_info = []

    for i, (start, end, rate) in enumerate(segments):
        frames = sample_video(video, (start, end), fps=rate)
        all_frames.extend(frames)
        segment_info.append({
            'segment_id': i,
            'range': (start, end),
            'sampling_rate': rate,
            'num_frames': len(frames)
        })

    # 2. 组合为单一批次
    stitched_context = {
        'frames': all_frames,
        'segment_info': segment_info
    }

    # 3. 跨片段感知
    verification = M_o.observe(
        frames=all_frames,
        question=question,
        mode='integrated',
        context=segment_info
    )

    return {
        'consistent': verification.is_consistent,
        'evidence': verification.cross_segment_evidence,
        'conclusion': verification.conclusion,
        'confidence': verification.confidence
    }
```

#### B.5 主体记忆表更新

```python
def update_entity_table(old_table, new_observation, scope):
    """
    更新主体记忆表

    Args:
        old_table: 现有实体表
        new_observation: 新观察结果
        scope: 观察范围（时间戳）

    Returns:
        new_table: 更新后的实体表
    """
    new_table = copy.deepcopy(old_table)

    # 1. 提取新实体
    new_entities = extract_entities(new_observation)

    # 2. 更新或创建实体条目
    for entity in new_entities:
        entity_id = entity.get('id') or f"entity_{len(new_table)+1}"

        if entity_id in new_table:
            # 更新现有实体
            existing = new_table[entity_id]
            existing['attributes'].update(entity.get('attributes', {}))
            existing['observations'].append({
                'timestamp': scope,
                'evidence': entity.get('evidence'),
                'confidence': entity.get('confidence')
            })
            existing['last_seen'] = scope['end']
        else:
            # 创建新实体
            new_table[entity_id] = {
                'id': entity_id,
                'type': entity.get('type'),
                'attributes': entity.get('attributes', {}),
                'first_seen': scope['start'],
                'last_seen': scope['end'],
                'observations': [{
                    'timestamp': scope,
                    'evidence': entity.get('evidence'),
                    'confidence': entity.get('confidence')
                }],
                'confidence': entity.get('confidence')
            }

    return new_table
```
