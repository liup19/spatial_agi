## 核心信息

### 摘要

本文提出了Visual Equivalence Reward Model (Visual-ERM)，一个多模态生成式奖励模型，用于vision-to-code任务。Vision-to-code任务要求模型将结构化视觉输入（如图表、表格、SVG）重构为可执行或结构化的表示，并保持高视觉保真度。

虽然最近的大型视觉语言模型（LVLMs）通过监督微调取得了强大的结果，但由于奖励信号错位，强化学习仍然具有挑战性。现有的奖励方法要么依赖文本规则，要么依赖粗糙的视觉嵌入相似性，这两种方法都无法捕获细粒度的视觉差异，且容易受到奖励黑客（reward hacking）的影响。

Visual-ERM是一个多模态生成式奖励模型，提供细粒度、可解释且与任务无关的反馈，直接在渲染的视觉空间中评估vision-to-code质量。集成到RL中后，Visual-ERM在chart-to-code上使Qwen3-VL-8B-Instruct提升了+8.4分，在table和SVG解析上平均提升了+2.7和+4.1分，并通过反思和修订进一步增强了测试时扩展（test-time scaling）。

作者还引入了VisualCritic-RewardBench (VC-RewardBench)，这是一个用于判断结构化视觉数据上细粒度图像到图像差异的基准。在该基准上，Visual-ERM（8B参数）显著优于Qwen3-VL-235B-Instruct，并接近领先的闭源模型。

结果表明，细粒度的视觉奖励监督对于vision-to-code RL是必要且充分的，无论任务特异性如何。

### 关键贡献

1. **提出Visual-ERM**: 一个多模态生成式奖励模型，为vision-to-code提供细粒度、可解释且与任务无关的奖励信号，可以无缝集成到RL管道和测试时扩展中。

2. **系统性分析奖励设计局限性**: 证明了基于文本的度量和视觉编码器相似性对于忠实的视觉评估都是不充分的。

3. **引入VC-RewardBench**: 一个评估结构化视觉数据（图表、表格、SVG）上细粒度图像到图像差异判断的基准。

4. **实验验证**: 证明Visual-ERM在多个vision-to-code任务中实现了有效的RL，使Qwen3-VL-8B-Instruct在chart-to-code上提升了+8.4分，在table-to-markdown和SVG-to-code上分别提升了+2.7和+4.1分；Visual-ERM还通过反思和修订增强了测试时扩展。

## 研究问题分析

### 1. 核心问题

Vision-to-code任务（如图表转代码、表格转Markdown、SVG转代码）是一个重要的新兴能力，但面临以下核心挑战：

**问题1: 奖励信号的设计困境**

现有的奖励方法分为两类，各有严重缺陷：

- **基于文本的奖励**: 如编辑距离和Tree Edit Distance Similarity (TEDS)
  - 完全在文本域中操作
  - 忽略关键的视觉线索，如对齐、间距和布局错误
  - 容易导致奖励黑客（reward hacking）
  - 存在模态偏差：只评估文本，不评估渲染后的视觉效果

- **基于视觉编码器的奖励**: 如DINO特征相似性
  - 是粗粒度的，受语义偏差影响
  - 对细粒度视觉细节不敏感，而这些细节对解析任务至关重要
  - 提供有限的可解释性
  - 优化代理目标而不是人类感知的视觉保真度

**关键洞察**: 这两种方法都提供脆弱、可被利用的奖励信号，不足以支持vision-to-code的RL训练。论文中给出一个例子：一个输出可以获得非常高的奖励分数（如DINO相似性0.99），但仍包含大量解析错误。

**问题2: 模态错位**

Vision-to-code任务的质量主要由视觉保真度决定，而不是文本本身。但现有奖励要么忽略视觉（文本规则），要么只使用粗糙的视觉相似性（视觉编码器），缺乏细粒度的跨模态评估。

**问题3: 任务特异性与通用性的权衡**

不同vision-to-code任务（图表、表格、SVG）有不同的失败模式和评估标准，但为每个任务单独设计奖励模型成本高昂。需要一个既能跨任务泛化，又能提供细粒度反馈的统一奖励模型。

**问题4: 可解释性需求**

有效的奖励不仅需要给出标量分数，还需要提供诊断性反馈，以指导测试时的反思和修订（test-time scaling）。现有的奖励方法缺乏这种可解释性。

### 2. 问题的重要性

**实际应用场景**:

- AI辅助前端开发（将UI设计转换为代码）
- 科学论文解析
- 知识管理和系统集成
- 数据可视化自动生成
- 矢量图形处理

**现有方法的局限性**:

- 监督微调（SFT）需要大规模、高质量的数据集，数据收集成本高
- 学习的模型缺乏跨域泛化能力
- RL作为有前景的替代方案，但缺乏可靠的奖励监督

### 3. 问题定义的合理性

**问题的科学价值**:

1. **填补空白**: 现有奖励模型主要针对文本生成（写作、对话），不支持视觉到代码的任务

2. **理论意义**: 探索多模态奖励建模的新范式，从文本域或粗粒度视觉域转向细粒度跨模态评估

3. **实践价值**: 推动vision-to-code技术的发展，降低下游应用的开发成本

**问题设定的完整性**:

论文不仅提出了问题，还：
- 系统分析了现有奖励设计的局限性
- 提供了定性和定量的证据（如图1展示了DINO奖励的语义偏差问题）
- 引入了新的评估基准来量化奖励模型质量
- 在多个任务上验证了方法的有效性

## 方法详解

### 3.1 Visual-ERM整体架构

Visual-ERM的完整流程包含三个相互关联的组件：

1. **奖励数据生成**: 通过控制的损坏和注释
2. **奖励模型的监督微调**
3. **集成到RL和测试时扩展**

此外，还引入了VisualCritic-RewardBench来基准测试细粒度的图像到图像差异判断。

#### 3.1.1 奖励数据生成

**核心思想**: 为训练细粒度的奖励模型，需要包含细粒度差异注释的图像对。

**步骤1: 数据收集**

从公开的vision-to-code数据集收集图像和对应的文本标注，涵盖图表、表格、SVG等领域：
- Chart-to-Code: 104K实例
- Table-to-Markdown: 125K实例
- SVG-to-Code: 111K实例

**步骤2: 构建损坏样本**

创建损坏的结构化输出y，通过两种互补的方式：

**(1) Edit（编辑方式）**:

使用强大的LVLMs（如GPT-5-mini、Qwen3-VL-235B-Instruct）扰动GT结构化文本，注入可控的、预定义的错误类型。

优势:
- 可以精确控制错误类型
- 覆盖预定义的错误模式
- 注意的错误分布更均匀

**(2) Infer（推理方式）**:

使用较弱的LVLMs（如Qwen3-VL-8B-Instruct）直接进行预测，从而采样自然发生的错误，更好地匹配实践中遇到的错误分布。

优势:
- 错误分布更真实
- 捕获模型实际推理时的失败模式
- 补充编辑方式的不足

**步骤3: 渲染与图像对构建**

对于每个损坏的结构化输出y，使用特定渲染器ℛ_m(·)将其渲染为图像Î = ℛ_m(y)，形成与真实图像I*的图像对(I*, Î)。

关键点:
- 使用不同任务的专用渲染器（图表、表格、SVG各有自己的渲染逻辑）
- 在视觉空间中评估，与人类视觉判断一致

#### 3.1.2 细粒度注释

**挑战**:

观察到一个重要差距：开源和闭源模型在定位图像差异方面存在显著差距。即使是非常大的模型，如Qwen3-VL-235B-Instruct，也无法可靠地定位细粒度差异。

**解决方案**: 采用蒸馏管道，将GPT-5-mini的强大差异定位能力转移到更高效的模型，实现可扩展的奖励生成。

**注释内容**:

每个图像对的细粒度注释a包含：
- 差异类型（type）
- 位置（location）
- 描述（description）
- 严重程度（severity）

这些注释将用于训练Visual-ERM生成类似的细粒度反馈。

**注释质量保证**:

- 使用最先进的专有模型（GPT-5-mini, Gemini2.5-Pro, Gemini3-Pro）
- 博士级注释者审查、修正和整合注释
- 最终筛选出高质量实例

#### 3.1.3 模型训练

**模型架构**:

在Qwen3-VL-8B-Instruct基础上构建Visual-ERM。虽然Qwen3-VL-8B-Instruct已经具有强大的多模态感知和推理能力，但作者发现它开箱即用不是一个可靠的图像到图像差异判断器，特别是对于结构化视觉，其中的关键差异通常是文本和布局中心的。

**训练数据**:

奖励数据集𝒟_reward = {(m, I*, Î, a)}
- m: 任务类型（Chart, Table, SVG）
- I*: 真实图像
- Î: 渲染的预测图像
- a: 细粒度差异注释

**优化目标**:

使用监督微调（SFT）训练Visual-ERM，优化负对数似然（NLL）目标：

```
ℒ(θ) = 𝔼_{(m,I*,Î,a)~𝒟_reward}[-log f_θ_ERM(a|x)]
```

其中x = (I*, Î)是输入图像对。

对于注释序列a = (a1, ..., aT)，token级目标为：

```
ℒ(θ) = 𝔼_{(m,I*,Î,a)}[-∑_{t=1}^T log f_θ_ERM(a_t|x, a_<t)]
```

其中a_<t = (a1, ..., a_{t-1})是前面的目标token。

**训练细节**:

- 学习率: 1×10^-5
- 批次大小: 32
- 训练轮数: 3个epoch
- 使用LLaMA-Factory框架实现

### 3.2 奖励设计与RL集成

#### 3.2.1 奖励设计

**奖励组成**:

Visual-ERM作为奖励模型，为RL提供细粒度反馈。奖励由两部分组成：

1. **Visual-ERM奖励 (r_verm)**: 基于细粒度差异的评估
2. **渲染成功奖励 (r_rsr)**: 鼓励可成功渲染的输出，类似于格式奖励

**差异预测**:

给定图像对(I*, Î)，Visual-ERM预测差异集合ℰ = {e_k}_{k=1}^K，每个差异e_k关联严重程度分数s_k ≥ 0。

**Severity Score计算**:

定义S_verm为预测严重程度的总和：

```
S_verm = ∑_{k=1}^K s_k
```

**归一化**:

为了将分数映射到[0, 1]范围，用当前任务𝒯中的最大严重程度分数归一化：

```
S̃_verm = S_verm / (max_{j∈𝒯} S_verm^{(j)} + ε)
```

其中ε是数值稳定性的小常数。

**奖励转换**:

转换为有界奖励：

```
r_verm = clip(1 - S̃_verm, 0, 1)
```

**总奖励**:

```
r = r_rsr + r_verm
```

其中r_rsr是渲染成功奖励（渲染成功为1，否则为0）。

**设计洞察**:

1. **多粒度评估**: 不仅给出标量分数，还提供细粒度的差异列表
2. **可解释性**: 每个差异都有类型、位置、描述和严重程度
3. **任务归一化**: 通过任务特定的最大分数进行归一化，确保跨任务的一致性
4. **二元奖励**: 使用clip函数确保奖励在[0, 1]范围内

#### 3.2.2 RL优化

**优化算法**: 使用基于GRPO（Group Relative Policy Optimization）的RL算法

**目标函数**:

最大化KL正则化的期望奖励：

```
max_θ 𝔼_{I*~𝒟}[
  𝔼_{y~π_θ(·|I*)}[r_rsr(ℛ_m(y)) + clip(1 - S̃_verm(I*, ℛ_m(y)), 0, 1)]
  - β·KL(π_θ(·|I*) || π_ref(·|I*))
]
```

其中:
- π_θ: 当前策略（参数为θ）
- π_ref: 参考策略（用于KL正则化）
- β: KL正则化强度
- 𝒟: 输入图像I*的训练分布

**训练设置**:

- 学习率: 1×10^-6
- 批次大小: 256
- 每个GRPO更新步骤采样8个rollouts以估计策略梯度
- 使用VERL框架实现

**RL流程**:

1. 给定输入图像I*
2. 策略模型π_θ生成结构化输出y
3. 使用渲染器ℛ_m(·)将y渲染为图像Î
4. 将(I*, Î)输入Visual-ERM，获得差异预测和严重程度分数
5. 计算奖励r
6. 使用GRPO更新策略

**关键优势**:

1. **端到端优化**: 直接在视觉空间中优化，与最终目标一致
2. **细粒度信号**: 比单一标量奖励提供更丰富的指导
3. **跨模态**: 融合视觉和文本证据
4. **可解释**: 可以分析哪些差异导致奖励变化

### 3.3 测试时扩展（Test-Time Scaling）

#### 3.3.1 反思与修订循环

Visual-ERM的另一个关键用例是为测试时扩展（TTS）提供反馈信号，通过迭代自我细化。

**流程**:

给定输入x = I*（原始图像）:

1. **初始预测**:
   ```
   y^(0) ~ π_θ(·|x)
   ```

2. **渲染**:
   ```
   Î^(0) = ℛ_m(y^(0))
   ```

3. **评估**:
   ```
   (r^(0), f^(0)) = f_θ_ERM(I*, Î^(0))
   ```
   其中r^(0)是估计的质量分数，f^(0)是细粒度描述。

4. **判断**:
   - 如果质量满意，停止
   - 如果质量不满意，进入修订

5. **修订**:
   ```
   y^(1) ~ π_θ(·|x, y^(0), f^(0))
   ```
   模型基于之前的预测和反馈进行修订。

6. **迭代**: 重复步骤2-5直到满意或达到最大迭代次数

**关键优势**:

1. **自我纠正**: 模型可以检测并修正自己的错误
2. **细粒度反馈**: 使用Visual-ERM提供的具体差异描述
3. **渐进式改进**: 多轮迭代逐步提升质量
4. **无需额外训练**: 仅在推理时使用

#### 3.3.2 TTS与RL的结合

TTS可以在RL训练的基础上进一步提升性能：

- **RL优化**: 在训练阶段改进策略
- **TTS优化**: 在推理阶段通过多轮自反思进一步提升

两者相辅相成：RL提供了更好的初始策略，TTS在此基础上进一步细化。

### 3.4 VisualCritic-RewardBench

#### 3.4.1 基准动机

现有奖励基准主要关注视觉-语言对齐（如VL-RewardBench），而用于细粒度图像到图像差异判别的基准仍然稀缺。因此作者引入了VC-RewardBench，专门针对结构化视觉（图表、表格、SVG），测量图像到图像的对齐保真度。

#### 3.4.2 基准构建

**候选池**: 4.5K候选图像对

**注释流程**:

1. **初步注释**: 使用三个最先进的专有模型：
   - GPT5-mini
   - Gemini-2.5-Pro
   - Gemini-3-Pro
   每个对获得三个独立注释

2. **人工整合与筛选**: 博士级注释者审查、修正和整合注释

3. **最终实例**: 1,335个高质量注释实例

**实例分布**:

- 图表: 595个实例
- 表格: 298个实例
- SVG: 442个实例

**差异类型分布**:

- **图表子集**:
  - text_error: 231个
  - style_error: 619个
  - data_error: 462个
  - structure_error: 62个

- **表格子集**:
  - text_error: 884个
  - numeric_error: 284个
  - layout_error: 185个

- **SVG子集**:
  - style_error: 151个
  - structure_error: 218个
  - shape_error: 360个
  - text_symbol_error: 12个

#### 3.4.3 评估协议

**评估挑战**:

VC-RewardBench要求输出结合结构化字段（如计数）和自由形式内容（如描述），因此精确匹配准确率不适用。

**解决方案**: 采用LLM-as-Judge协议

**具体步骤**:

1. **匹配**: 判定LLM将预测的差异A^与真实注释A*匹配，识别TP/FP/FN

2. **计算指标**:
   - Precision (精确率)
   - Recall (召回率)
   - F1 Score (F1分数)

3. **两个F1变体**:
   - **F1_hard (F1_h)**: 严格匹配标准，需要紧密的语义和属性级一致性
   - **F1_soft (F1_s)**: 软匹配标准，允许更宽松的语义对齐

4. **严重程度评分**: 由于模型还输出每个差异的严重程度分数，将严重程度求和，并报告与真实总分的Pearson相关系数，记为S_c，测量整体评分一致性。

**基准设计洞察**:

1. **结构化+非结构化**: 结合了两种类型的评估，既测量结构化准确性，也评估描述质量
2. **细粒度**: 不仅测量是否正确识别差异，还评估严重程度
3. **多维度**: 通过Precision/Recall/F1和S_c多个指标全面评估
4. **高质量**: 通过人工整合确保注释质量

## 实验分析

### 4.1 实验设置

#### 4.1.1 模型配置

**Visual-ERM（奖励模型）**:

- 基础模型: Qwen3-VL-8B-Instruct
- 训练数据:
  - Chart-to-Code: 104K实例
  - Table-to-Markdown: 125K实例
  - SVG-to-Code: 111K实例
- 训练设置:
  - 学习率: 1×10^-5
  - 批次大小: 32
  - 训练轮数: 3 epochs

**策略模型（用于RL）**:

1. **Qwen3-VL-8B-Instruct**:
   - 默认策略模型，跨任务使用
   - 使用Visual-ERM作为奖励模型进行RL训练

2. **VinciCoder-8B-SFT**:
   - 更强的面向解析的策略模型
   - 在Chart-to-Code和SVG-to-Code上作为替代策略主干

3. **Visual-ERM的优势**:
   - 显式整合视觉感知与跨模态grounding和推理
   - 使用视觉结构和渲染文本评估重构

### 4.9 不同判别器模型的消融

**评估协议挑战**:

VC-RewardBench评估奖励模型识别细粒度图像到图像差异和vision-to-code重构错误的能力。除了客观指标（如标量分数、类型和严重程度）外，VC-RewardBench还包括自由形式的差异描述。由于这些描述是开放式的，可能使用不同的措辞来指代相同的潜在问题，精确字符串匹配不足以进行可靠的评估。

**LLM辅助匹配协议**:

为了以可扩展和一致的方式评估此类自由形式输出，采用LLM-as-Judge来执行奖励模型的预测错误列表与注释错误列表之间的错误级对齐。

**具体步骤**:

给定预测错误和注释错误，判定为每个预测项目确定它是否：
1. 正确匹配真实错误
2. 是幻觉/不受支持的主张
3. 对应于存在但描述不正确的错误

对称地，判定还识别被模型遗漏的真实错误。基于此对齐，计算两个F1变体：

1. **严格匹配标准 (F1_h, F1_hard)**:
   - 需要紧密的语义和属性级一致性

2. **软匹配标准 (F1_s, F1_soft)**:
   - 允许更宽松的语义对齐

**关键洞察**:

这种评估协议确保了：
1. **公平评估**: 不同措辞但语义相同的描述不会受到不公平惩罚
2. **细粒度**: 不仅测量是否识别错误，还测量描述的准确性
3. **可扩展**: 使用LLM-as-Judge可以处理大规模评估
4. **一致性**: 比人工评估更一致

## 局限性分析

### 1. 渲染器依赖性

**问题描述**:

Visual-ERM的设计依赖将代码渲染为图像的步骤。这意味着：

1. **任务依赖**: 每个任务（Chart, Table, SVG）需要专用渲染器
2. **渲染器质量**: 奖励信号的质量受限于渲染器的质量和能力
3. **渲染失败**: 如果渲染失败（如代码语法错误），整个评估流程中断
4. **渲染开销**: 渲染步骤增加了计算开销，影响RL训练和推理速度

**论文的应对**:

- 引入渲染成功奖励（r_rsr）来鼓励可成功渲染的输出
- 这在某种程度上缓解了渲染失败的问题，但没有根本解决

**潜在问题**:

1. **渲染器差异**: 不同任务的渲染器可能有不同的能力和限制，导致奖励信号不完全可比
2. **渲染歧义**: 某些代码可能有多种渲染方式，导致不确定性
3. **渲染器更新**: 随着渲染器的演进，需要重新训练或适应奖励模型

### 2. 数据质量和多样性

**问题描述**:

虽然论文使用强大的专有模型（GPT-5-mini, Gemini）生成训练数据，但仍然存在潜在问题：

1. **专有模型偏差**: 训练数据反映了GPT-5-mini等的偏差和局限
2. **错误模式覆盖**: Edit和Infer方法可能无法覆盖所有可能的错误模式
3. **长尾错误**: 真实世界中的罕见或复杂错误可能未被充分表示
4. **分布偏移**: 训练数据中的错误分布可能与实际部署时的错误分布不同

**论文的应对**:

- 结合Edit（受控错误）和Infer（自然错误）两种方法
- 使用多个专有模型进行注释，然后人工整合
- 大规模数据集（总计340K实例）

**潜在问题**:

1. **人工整合成本**: 博士级注释者整合4.5K候选对的过程昂贵且难以扩展
2. **长尾分布**: 即使340K实例，可能仍然无法覆盖某些罕见错误
3. **领域适应性**: 训练数据主要来自公开数据集，可能不适应特定领域的vision-to-code任务

### 3. 标量奖励的信息损失

**问题描述**:

Visual-ERM虽然生成细粒度的差异列表，但最终转换为标量奖励用于RL优化：

1. **信息压缩**: 将多维差异信息压缩为单一标量值，丢失细粒度细节
2. **加权和挑战**: 将多个差异的严重程度求和，隐含假设所有差异同等重要或可以线性组合
3. **归一化挑战**: 使用任务内的最大分数归一化，但对不同任务可能不完全可比

**论文的应对**:

- 在TTS中直接使用细粒度反馈，避免转换为标量
- 在RL中使用标量奖励但保持可解释性（差异列表仍可用于分析）

**潜在问题**:

1. **优化复杂性**: 如果直接使用细粒度反馈优化，RL算法将更复杂
2. **权重学习**: 不同差异类型的权重应该如何确定？论文没有明确讨论
3. **动态归一化**: 使用任务内的最大分数归一化，但在训练过程中最大分数会变化，可能导致归一化不稳定

### 4. 计算开销

**问题描述**:

Visual-ERM的训练和使用需要显著的计算资源：

1. **训练开销**: 需要在340K实例上训练8B模型，需要大量GPU资源
2. **推理开销**: RL训练的每个步骤需要渲染和Visual-ERM推理，增加训练时间
3. **TTS开销**: 多轮反思每次都需要渲染和Visual-ERM评估，增加推理延迟

**论文的应对**:

- 没有明确讨论计算开销的优化
- 暗示使用8B而不是更大的模型是为了平衡性能和效率

**潜在问题**:

1. **可扩展性**: 对于更大的模型（如70B）或更多任务，计算成本可能成为瓶颈
2. **实时应用**: 对于实时或交互式应用，TTS的多轮反思可能延迟过高
3. **资源需求**: 普通研究团队或小公司可能无法承担Visual-ERM训练所需的计算资源

### 5. 任务范围限制

**问题描述**:

Visual-ERM专注于三种结构化视觉任务（Chart, Table, SVG），但vision-to-code的范围更广：

1. **UI设计**: 将Figma/Sketch设计转换为代码
2. **文档布局**: 将扫描文档或PDF转换为结构化表示
3. **3D场景**: 将3D渲染图像转换为3D建模代码
4. **动画**: 将动画视觉转换为动画代码

**论文的应对**:

- 声称方法可以扩展到其他vision-to-code任务
- 提供了框架（数据生成、注释、训练）的通用流程

**潜在问题**:

1. **任务差异性**: 不同vision-to-code任务的失败模式可能差异很大，需要重新设计错误类型
2. **渲染器可用性**: 某些任务可能没有可靠的渲染器，或不适用渲染范式
3. **泛化能力**: 虽然Visual-ERM在Chart/Table/SVG之间显示跨任务泛化，但不保证能泛化到完全不同的任务

### 6. 与下游集成的挑战

**问题描述**:

论文主要关注vision-to-code任务的评估和优化，但讨论与下游应用的集成较少：

1. **端到端流程**: Visual-ERM如何集成到完整的开发流程中？
2. **用户反馈**: 除了Visual-ERM的自动评估，如何结合用户反馈？
3. **多模态输入**: 如果输入不仅仅是视觉（如用户说明、上下文），如何评估？

**论文的应对**:

- 提到了AI辅助前端开发、科学论文解析等应用场景
- 但没有详细讨论集成策略

**潜在问题**:

1. **反馈循环**: 在真实应用中，需要结合用户反馈和自动评估，如何设计这个循环？
2. **多约束优化**: 真实应用中可能有多个约束（如代码可读性、性能），如何与视觉保真度平衡？
3. **可维护性**: 生成代码的可维护性在真实项目中很重要，Visual-ERM目前不评估这个维度

### 7. 评估基准的局限性

**问题描述**:

VC-RewardBench是一个有价值的贡献，但仍有一些局限性：

## 与Spatial AGI的相关性评估

### 1. Vision-to-Code与Spatial AGI的关系

#### 1.1 直接贡献

**Vision-to-Code作为Spatial AGI的核心能力**:

Vision-to-code是Spatial AGI的一个重要组成部分，因为它涉及：

1. **视觉理解**: 理解结构化视觉输入（图表、表格、UI设计）
2. **空间关系**: 识别视觉元素的空间布局和对齐
3. **抽象表示**: 将视觉转换为抽象的结构化表示（代码、标记语言）
4. **生成能力**: 基于视觉输入生成可执行的代码

这些能力都是Spatial AGI的核心需求：
- **机器人**: 理解环境视觉并生成行动计划或代码
- **AR/VR**: 将视觉设计转换为可渲染的代码
- **智能助手**: 解析视觉信息并执行相应操作

**Visual-ERM的提升**:

Visual-ERM通过提供细粒度、可解释、任务无关的奖励信号，显著改进了vision-to-code的质量：

- **Chart-to-Code**: +8.4提升
- **Table-to-Markdown**: +2.7提升
- **SVG-to-Code**: +4.1提升

这些提升直接转化为Spatial AGI在相关任务上的更好表现。

#### 1.2 技术启发

**细粒度视觉评估**:

Visual-ERM的核心创新是提供细粒度的视觉评估，这对Spatial AGI有重要启发：

1. **多尺度感知**:
   - Visual-ERM同时考虑全局结构和局部细节
   - Spatial AGI也需要在不同尺度上理解空间（全局布局、局部细节）

2. **跨模态推理**:
   - Visual-ERM结合视觉和文本证据进行评估
   - Spatial AGI需要结合视觉、语言、动作等多模态信息

3. **可解释反馈**:
   - Visual-ERM提供差异类型、位置、严重程度等结构化反馈
   - Spatial AGI需要类似的机制来提供可解释的空间推理反馈

**任务无关的奖励建模**:

Visual-ERM展示了如何设计跨任务的奖励模型：

1. **共享失败模式**:
   - Chart、Table、SVG共享一些失败模式（如文本识别、布局对齐）
   - Spatial AGI的不同任务（导航、操作、对话）也可能共享一些失败模式

2. **跨任务泛化**:
   - Visual-ERM在一个任务上训练可以泛化到其他任务
   - Spatial AGI可以设计跨任务的奖励模型，提高学习效率

**生成式奖励模型**:

Visual-ERM使用生成式模型来生成奖励，这为Spatial AGI提供了新的思路：

1. **丰富的反馈**:
   - 不仅提供标量奖励，还提供结构化差异描述
   - Spatial AGI可以生成丰富的环境反馈（位置、状态、约束等）

2. **推理能力**:
   - Visual-ERM的多模态推理能力用于评估
   - Spatial AGI可以使用类似的推理能力来理解和评估空间状态

#### 1.3 应用场景

**机器人视觉执行**:

1. **场景理解**: 机器人理解场景视觉（如货架、物品布局）并生成操作代码
   - Visual-ERM可以评估生成的操作计划与场景视觉的保真度
   - 细粒度反馈可以指出哪些操作步骤或识别错误

2. **视觉指令理解**: 机器人从视觉演示或图表中理解任务指令
   - Visual-ERM可以评估对视觉指令的理解准确性
   - 检测细微的误解或遗漏

3. **空间验证**: 机器人验证操作后的结果是否符合预期
   - Visual-ERM可以比较操作前后的视觉状态
   - 提供细粒度的变化反馈

**AR/VR内容生成**:

1. **UI设计到代码**: 将Figma/Sketch设计转换为AR/VR代码
   - Visual-ERM可以扩展到UI视觉的细粒度评估
   - 检测布局、样式、交互元素等差异

2. **3D场景重构**: 从2D图像生成3D场景描述
   - Visual-ERM的概念可以扩展到3D视觉评估
   - 评估生成3D模型与输入图像的视觉保真度

3. **实时渲染优化**: 在AR/VR中优化渲染效果
   - Visual-ERM的细粒度反馈可以指导渲染参数调整
   - 在测试时使用TTS进行迭代优化

**智能文档处理**:

1. **文档解析**: 从扫描文档或PDF提取结构化信息
   - Visual-ERM可以评估解析结果的视觉保真度
   - 检测文本识别、表格结构、图像嵌入等错误

2. **数据可视化生成**: 从数据生成可视化图表
   - Visual-ERM可以评估生成图表与数据的匹配度
   - 检测数据错误、样式不一致、布局问题

3. **科学图表理解**: 理解科学论文中的复杂图表
   - Visual-ERM可以评估对图表内容的理解准确性
   - 检测对数据、趋势、异常值的理解错误

### 2. 可解释的奖励信号

#### 2.1 可解释性的重要性

**Spatial AGI的可解释性需求**:

Spatial AGI需要在复杂、高风险的环境中运行，可解释性至关重要：

1. **安全性**: 机器人需要解释其动作决策，确保不会造成危险
2. **调试**: 开发者需要理解系统为什么做出特定决策，以便调试和改进
3. **人机协作**: 人类需要理解AI的空间推理，以便有效协作
4. **合规性**: 某些应用领域（如医疗、工业）需要可解释的决策过程

**Visual-ERM的可解释性优势**:

Visual-ERM提供了高度可解释的奖励信号：

1. **结构化差异列表**: 每个差异都有类型、位置、描述、严重程度
2. **视觉定位**: 可以定位到具体的视觉区域或元素
3. **自然语言描述**: 用自然语言描述差异，易于人类理解
4. **严重程度量化**: 对每个差异进行严重程度评分

#### 2.2 对Spatial AGI的启发

**多粒度反馈**:

Spatial AGI可以设计类似的多粒度反馈系统：

1. **任务级别**: 整体任务是否成功？
2. **步骤级别**: 哪些步骤有问题？
3. **细节级别**: 具体哪个动作或状态有问题？
4. **解释级别**: 为什么有问题？如何改进？

**示例**: 机器人抓取任务
- 任务级别: 抓取成功/失败
- 步骤级别: 定位、接近、抓握三个步骤中哪个失败
- 细节级别: 定位误差5mm，接近速度过快
- 解释级别: 视觉遮挡导致定位不准确，需要调整相机角度

**细粒度诊断**:

Visual-ERM的差异类型为Spatial AGI提供了细粒度诊断的模板：

1. **空间关系错误**: 位置、对齐、距离等空间关系
2. **对象识别错误**: 误识别、遗漏、分类错误
3. **结构错误**: 整体布局、层次、拓扑结构
4. **属性错误**: 颜色、大小、形状等视觉属性
5. **内容错误**: 文本、数字、符号等内容识别

**示例**: AR导航中的空间错误诊断
- 空间关系错误: 路径与实际道路偏移
- 对象识别错误: 将路标误识别为广告牌
- 结构错误: 建筑物层次结构错误
- 属性错误: 道路宽度估算错误
- 内容错误: 路标文字识别错误

**自适应反馈**:

Visual-ERM可以根据上下文调整反馈的粒度和侧重点，这对Spatial AGI很有启发：

1. **任务相关性**: 不同任务关注不同的错误类型
2. **用户偏好**: 用户可能更关注某些类型的错误
3. **历史上下文**: 基于历史错误调整反馈策略
4. **环境约束**: 在资源受限时简化反馈

**示例**: 智能家居助手
- 任务相关性: 烹饪任务更关注安全，清洁任务更关注彻底性
- 用户偏好: 用户A更喜欢详细反馈，用户B更喜欢简洁反馈
- 历史上下文: 如果用户经常犯某类错误，加强该类错误反馈
- 环境约束: 在紧急情况下简化反馈，快速执行

#### 2.3 实现策略

**生成式判别器**:

Visual-ERM使用生成式模型作为判别器，Spatial AGI可以采用类似策略：

1. **多模态理解**: 同时处理视觉、语言、空间等输入
2. **结构化输出**: 生成结构化的错误报告或反馈
3. **推理能力**: 通过推理判断哪些错误是关键的
4. **自然语言生成**: 用自然语言解释问题

**架构设计**:

```
输入: (环境状态, 期望状态, 行动序列)
  ↓
多模态编码器 (视觉、语言、空间)
  ↓
差异检测模块
  ↓
差异分类 (类型、严重程度)
  ↓
差异定位 (空间位置)
  ↓
差异描述生成 (自然语言)
  ↓
输出: 结构化错误报告
```

**训练数据**:

类似于Visual-ERM，Spatial AGI需要构建细粒度的反馈数据集：

1. **错误类型定义**: 定义空间相关的错误类型
2. **合成错误**: 在正确操作上注入可控错误
3. **自然错误**: 收集真实系统的错误案例
4. **细粒度注释**: 使用人工或专有模型进行细粒度注释

**评估基准**:

类似于VC-RewardBench，Spatial AGI需要细粒度评估基准：

1. **场景库**: 涵盖不同类型空间场景
2. **错误案例**: 每个场景的错误样本
3. **细粒度标注**: 错误类型、位置、严重程度
4. **评估协议**: LLM-as-Judge或人工评估

### 3. 细粒度、任务无关的反馈

#### 3.1 细粒度的价值

**细粒度vs粗粒度**:

传统的奖励信号往往是粗粒度的标量值，而Visual-ERM展示了细粒度反馈的价值：

**粗粒度反馈**:
- 优点: 简单、易计算、易优化
- 缺点: 信息损失、难以诊断、优化方向不明确

**细粒度反馈**:
- 优点: 信息丰富、可诊断、优化方向明确
- 缺点: 复杂、计算开销大、优化算法复杂

**Spatial AGI中的细粒度需求**:

Spatial AGI尤其需要细粒度反馈，因为：

1. **空间复杂性**: 空间任务涉及多个维度（位置、方向、速度、力等）
2. **多步骤**: 大多数空间任务是多步骤的，需要知道哪一步出问题
3. **安全关键**: 空间任务往往涉及安全，需要精确的反馈避免危险
4. **调试需求**: 空间任务调试困难，细粒度反馈能加速调试

**示例**: 机器人装配任务

**粗粒度反馈**:
- 成功/失败 (0/1)
- 无法知道失败原因

**细粒度反馈**:
- 步骤1 (定位): 误差5mm
- 步骤2 (接近): 角度偏差3度
- 步骤3 (抓握): 力度不足
- 步骤4 (装配): 对齐误差2mm

这种细粒度反馈可以：
- 快速定位问题
- 指导具体改进
- 避免重复错误
- 优化学习过程

#### 3.2 任务无关的泛化

**任务无关vs任务相关**:

Visual-ERM展示了任务无关奖励模型的潜力：

**任务相关奖励**:
- 优点: 针对性强，性能可能更好
- 缺点: 需要为每个任务单独设计，成本高

**任务无关奖励**:
- 优点: 一套方法适用多任务，效率高
- 缺点: 可能不如任务特定的方法性能好

**Visual-ERM的平衡**:

Visual-ERM通过以下方式平衡任务无关性和性能：

1. **共享失败模式**: Chart、Table、SVG共享文本识别、布局对齐等失败模式
2. **多任务训练**: 在多任务数据上训练，学习共享特征
3. **任务特定归一化**: 使用任务特定的最大分数归一化，保持一致性
4. **细粒度差异**: 即使是任务无关，也能捕获任务特定的细节

**Spatial AGI中的任务无关性**:

Spatial AGI同样可以从任务无关的奖励模型中受益：

**共享的空间失败模式**:

1. **导航任务**: 位置估计、路径规划、障碍物检测
2. **操作任务**: 抓取定位、力度控制、接触检测
3. **协作任务**: 人机距离、动作协调、意图理解

这些任务共享一些空间推理能力，可以设计任务无关的奖励模型。

**跨任务学习**:

- 在导航任务上学习的奖励模型可以泛化到操作任务
- 通过多任务训练提高泛化能力
- 减少为新任务设计奖励的成本

**领域适应性**:

虽然奖励模型是任务无关的，但可以通过以下方式适应特定领域：

1. **领域微调**: 在少量领域数据上微调
2. **领域提示**: 通过prompt engineering适应领域
3. **领域归一化**: 使用领域特定的归一化策略

#### 3.3 混合奖励设计

**标量+细粒度**:

Visual-ERM虽然生成细粒度差异，但在RL中使用时转换为标量奖励。Spatial AGI可以设计更灵活的奖励：

**方案1: 纯标量奖励**
- 简单，但信息损失
- 类似于Visual-ERM在RL中的使用

**方案2: 纯细粒度奖励**
- 信息丰富，但优化复杂
- 需要设计新的RL算法

**方案3: 混合奖励**
- 主要使用标量奖励进行优化
- 使用细粒度反馈进行诊断和调试
- 在TTS中使用细粒度反馈

**方案4: 层次化奖励**
- 不同层次使用不同粒度的奖励
- 任务层次：粗粒度
- 步骤层次：中等粒度
- 动作层次：细粒度

**推荐**: Spatial AGI采用方案3或4，在优化效率和反馈质量之间找到平衡。

**多维度奖励**:

除了细粒度，奖励还可以从多个维度评估：

1. **视觉保真度**: 类似于Visual-ERM，评估视觉一致性
2. **功能正确性**: 评估任务是否正确完成
3. **效率**: 评估时间和资源消耗
4. **安全性**: 评估安全约束
5. **可解释性**: 评估决策的可解释性

Spatial AGI需要综合这些维度设计奖励函数。

**自适应奖励**:

根据任务阶段和上下文自适应调整奖励：

1. **训练阶段**: 初期使用粗粒度奖励快速学习，后期使用细粒度奖励精细调优
2. **探索阶段**: 使用宽松的奖励鼓励探索
3. **利用阶段**: 使用严格奖励确保性能
4. **安全场景**: 加强安全相关的奖励

## 技术发现

### 1. 多模态奖励建模的新范式

**发现**:

Visual-ERM提出了一个多模态奖励建模的新范式，与传统方法有本质区别：

**传统范式**:

1. **文本规则**: 纯文本空间评估（如TEDS、编辑距离）
2. **视觉编码器**: 粗粒度视觉特征相似性（如DINO）
3. **单模态**: 只在文本域或视觉域中操作
4. **标量奖励**: 输出单一标量值

**Visual-ERM范式**:

1. **跨模态**: 融合视觉和文本证据
2. **细粒度**: 捕获细微视觉细节
3. **生成式**: 生成结构化差异描述
4. **可解释**: 提供类型、位置、严重程度等多维度信息

**关键洞察**:

多模态奖励建模需要：
- 跨模态理解能力
- 细粒度感知能力
- 结构化生成能力
- 推理能力

这些能力超越了传统的相似性计算，需要更深的多模态理解。

**验证**:

实验证明，Visual-ERM范式显著优于传统方法：
- Chart-to-Code: +8.4 vs DINO-based
- Table-to-Markdown: +2.7 vs TEDS/DINO
- SVG-to-Code: +4.1 vs DINO

### 2. 针对性训练 vs 简单缩放

**发现**:

在细粒度差异判断任务上，针对性训练的8B模型显著优于简单缩放的235B模型。

**数据支持**:

- Qwen3-VL-235B-Instruct (235B): 在VC-RewardBench上表现挣扎
- Visual-ERM (8B): 基于相同的8B模型，但针对性训练后：
  - F1_h: +36.8
  - F1_s: +38.2
  - S_c: +40.9

**原因分析**:

1. **任务不匹配**: 通用LVLM训练目标与细粒度差异判断任务不匹配
2. **数据不匹配**: 通用LVLM的训练数据不包含细粒度差异标注
3. **能力不匹配**: 细粒度差异判断需要特定的能力，通用LVLM没有充分发展这些能力

**启示**:

1. **专业化价值**: 专业化训练的价值不在于模型大小，而在于任务对齐
2. **数据质量**: 高质量的任务特定数据比低质量的通用数据更有价值
3. **能力聚焦**: 针对特定能力训练比全能训练更有效

**对Spatial AGI的启示**:

Spatial AGI不应仅仅追求更大的模型，而应该：
- 针对空间推理能力进行专业化训练
- 构建高质量的空间相关数据集
- 设计针对性的评估和训练流程

### 3. 多任务正迁移

**发现**:

在Visual-ERM上，多任务训练展示了正迁移效应，而不是负迁移。

**数据支持**:

**单任务训练**:
- Chart-Data-Only: Chart好 (F1_h=40.1)，Table差 (F1_h=17.2)，SVG差 (F1_h=11.3)
- Table-Data-Only: Table好 (F1_h=53.6)，Chart差 (F1_h=17.1)
- 强烈的负迁移或零迁移

**多任务训练** (Mix-Data):
- Chart: F1_h=39.9 (略低于Chart-only)
- Table: F1_h=56.4 (高于Table-only)
- SVG: F1_h=28.3 (显著高于SVG-only)
- 整体: 最佳平均性能 (F1_h=42.1)

**原因分析**:

1. **共享失败模式**: Chart、Table、SVG共享文本识别、布局对齐等失败模式
2. **互补数据**: 不同任务的数据互补，提供更全面的训练
3. **泛化能力**: 多任务训练提高泛化能力，减少过拟合

**对RL的影响**:

多任务训练的RM在实际RL训练中也表现更好：
- Table-only RM: +1.8提升
- Mix-Data RM: +2.7提升

**启示**:

1. **跨任务泛化**: 即使任务看似不同，只要共享一些底层能力，就能从多任务训练中受益
2. **数据多样性**: 数据多样性的价值在于覆盖更全面的错误模式
3. **平衡优化**: 多任务训练在保持任务内性能的同时提高跨任务泛化

**对Spatial AGI的启示**:

Spatial AGI的不同任务（导航、操作、协作等）也可能共享空间推理能力，因此：
- 可以设计跨任务的奖励模型
- 多任务训练可以提高泛化能力
- 降低为新任务设计奖励的成本

### 4. 测试时扩展的有效性

**发现**:

Visual-ERM不仅用于RL训练，还在测试时扩展（TTS）中有效。

**数据支持**:

**无RL训练的基线**:
- TTS (3轮反思): +8.0提升

**有RL训练的模型**:
- TTS (3轮反思): +3.1提升

**关键观察**:

1. **独立于RL有效**: TTS即使没有RL训练也显著有效
2. **与RL互补**: 在RL基础上TTS还能带来额外提升
3. **迭代改进**: 多轮反思逐步提升质量

**机制分析**:

Visual-ERM的细粒度反馈使TTS能够：

1. **自我检测**: 模型可以检测自己的错误
2. **自我纠正**: 基于细粒度反馈进行针对性修改
3. **渐进改进**: 每轮迭代基于前一轮的反馈

**启示**:

1. **反馈循环**: 细粒度反馈可以创建自我改进的反馈循环
2. **训练-推理分离**: 训练时的细粒度反馈可以迁移到推理时
3. **无需额外标注**: TTS完全依赖自动反馈，无需人工标注

**对Spatial AGI的启示**:

Spatial AGI可以采用类似的TTS策略：

1. **执行-评估-反思-修正**: 在测试时循环这个过程
2. **多轮迭代**: 逐步改进空间推理和动作执行
3. **自动化**: 完全自动，无需人工干预

### 5. 渲染范式的有效性

**发现**:

将代码渲染为图像并在视觉空间评估是一个有效的范式。

**与传统方法的对比**:

**文本域评估**:
- 优点: 简单、快速
- 缺点: 忽略视觉线索、容易奖励黑客

**视觉特征评估**:
- 优点: 考虑视觉信息
- 缺点: 粗粒度、缺乏可解释性

**渲染后视觉评估**:
- 优点: 直接评估最终视觉效果、细粒度、可解释
- 缺点: 需要渲染器、计算开销大

**关键洞察**:

Vision-to-code任务的最终目标是视觉保真度，因此直接在视觉空间评估是最直接和有效的方法。

**验证**:

实验证明，Visual-ERM（渲染范式）显著优于文本规则和视觉编码器相似性。

**适用条件**:

渲染范式适用于：
1. 有可靠的渲染器
2. 视觉保真度是主要目标
3. 可以承受渲染的计算开销

**对Spatial AGI的启示**:

Spatial AGI中很多任务也有类似的"执行-验证"范式：

1. **动作执行**: 执行动作（如抓取）
2. **结果渲染**: 渲染结果（如抓取后的场景）
3. **视觉评估**: 评估结果与预期的视觉一致性
4. **细粒度反馈**: 指出具体哪些动作或状态有问题

### 6. 细粒度vs粗粒度的权衡

**发现**:

细粒度反馈虽然复杂，但在RL和TTS中都带来显著收益。

**收益分析**:

**RL训练**:
- +8.4 (Chart), +2.7 (Table), +4.1 (SVG) vs 基线奖励
- 细粒度反馈提供更丰富的梯度信息

**TTS**:
- +8.0 (基线), +3.1 (RL) vs 无TTS
- 细粒度反馈指导迭代改进

**成本分析**:

1. **计算开销**: 细粒度反馈需要更多计算
2. **数据标注**: 需要细粒度的标注数据
3. **算法复杂**: 优化算法更复杂

**权衡**:

虽然细粒度反馈有额外成本，但收益显著超过成本，特别是在性能关键的任务中。

**对Spatial AGI的启示**:

Spatial AGI需要在细粒度和粗粒度之间找到平衡：

1. **训练阶段**: 可以使用细粒度反馈
2. **推理阶段**: 可以使用更粗粒度的反馈以加快速度
3. **关键任务**: 使用细粒度反馈以保障安全和性能
4. **调试阶段**: 使用细粒度反馈以快速定位问题

### 1. 最有趣的发现

**发现**: 针对性训练的8B模型在细粒度差异判断上显著优于简单缩放的235B模型

**原因**: 这个发现打破了"更大模型总是更好"的常见假设，表明任务对齐和专业化训练比简单的模型缩放更重要

**深入思考**:

这个发现对整个AI领域有重要启示：

1. **专业化分工**: 未来的AI系统可能不是单一的大型模型，而是多个专业化模型的集合
   - Visual-ERM: 专门评估视觉差异
   - 代码模型: 专门生成代码
   - 语言模型: 专门处理语言理解
   - 每个模型在自己的领域达到最优，通过协作完成复杂任务

2. **任务导向的模型设计**: 模型设计应该从"我们要什么能力"而不是"我们要多大模型"开始
   - 首先明确任务需求（如细粒度差异判断）
   - 然后设计针对性的架构和训练流程
   - 最后选择合适的模型规模

3. **数据质量>模型大小**: 高质量的任务特定数据比低质量的通用数据更有价值
   - Visual-ERM的340K细粒度标注数据比235B模型的训练数据更有针对性
   - 这启示我们，数据集的质量和相关性比规模更重要

**对Spatial AGI的启示**:

Spatial AGI的发展也应该遵循这个原则：

1. **专业化空间模型**: 设计专门处理空间推理的模型，而不是依赖通用的多模态模型
2. **任务导向**: 明确空间推理的具体需求（如3D理解、动作规划、人机协作），然后针对性设计
3. **数据驱动**: 收集高质量的空间相关数据（如3D场景、动作序列、协作交互）

### 2. 最意外的结果

**结果**: DINO-based奖励在Table-to-Markdown任务上不仅没有帮助，反而导致性能下降

**原因**: DINO特征主要捕获语义信息和全局外观，但表格任务主要由精确的文本内容和布局约束主导

**深入思考**:

这个结果揭示了几个关键问题：

1. **代理目标的陷阱**: DINO-based奖励优化的是一个代理目标（特征相似性），而不是真实目标（视觉保真度）
   - 类似于"古德哈特定律": 当一个度量成为目标时，它就不再是一个好的度量
   - DINO特征相似性与人类感知的表格质量不相关
   - 优化DINO相似性可能损害表格的精确性和可用性

2. **任务特性的重要性**: 不同任务有不同的评估维度和关注点
   - 表格: 精确文本、数值准确性、布局对齐
   - 图表: 整体趋势、数据关系、样式一致性
   - SVG: 几何形状、路径精度、样式细节
   - DINO无法区分这些差异，一刀切的相似性不适用

3. **模态偏差的风险**: 单一模态的评估可能引入系统性偏差
   - DINO是纯视觉的，忽略文本内容
   - TEDS是纯文本的，忽略视觉布局
   - 真正的评估需要跨模态

**对Spatial AGI的启示**:

Spatial AGI中也需要警惕类似的代理目标陷阱：

1. **避免过度简化**: 不要用简单的度量替代复杂的目标
   - 如：只用位置误差评估抓取质量，忽略了力度、姿态等因素
   - 应该设计多维度的评估指标

2. **任务特定评估**: 针对不同类型的空间任务设计不同的评估方法
   - 导航: 路径效率、安全性、时间
   - 操作: 精度、力度、稳定性
   - 协作: 人机距离、协调性、意图理解

3. **跨模态评估**: 空间任务往往涉及视觉、触觉、语言等多模态信息
   - 评估需要综合考虑多个模态
   - 类似于Visual-ERM的跨模态评估方法

### 3. 最有价值的启发

**启发**: 细粒度、可解释的奖励信号是提升RL效果的关键

**深入思考**:

这个启发对强化学习有广泛的意义：

1. **RL的瓶颈不是算法，而是奖励设计**:
   - 现代RL算法（如PPO、GRPO）已经相当成熟
   - 真正的挑战在于设计好的奖励信号
   - 细粒度奖励提供更丰富的学习信号

2. **可解释性不仅对人类重要，对机器也很重要**:
   - 细粒度奖励让机器理解"为什么得到这个奖励"
   - 结构化反馈指导机器进行有针对性的改进
   - 类似于人类需要详细反馈来改进技能

3. **奖励设计的艺术**:
   - 好的奖励设计是艺术和科学的结合
   - 需要理解任务的本质和失败模式
   - 需要平衡细粒度和可计算性
   - Visual-ERM提供了一个很好的范式

**对Spatial AGI的启示**:

Spatial AGI的RL训练也依赖于精细的奖励设计：

1. **细粒度空间奖励**: 设计细粒度的空间相关奖励
   - 位置、姿态、速度、力等多维度
   - 每个维度提供有针对性的反馈

2. **可解释的空间反馈**:
   - 不仅仅给出"成功/失败"
   - 而是解释"为什么失败"、"如何改进"
   - 例如："抓取力度不足，增加20%力度"

3. **层次化奖励**:
   - 任务层次: 整体任务完成度
   - 步骤层次: 每个步骤的质量
   - 动作层次: 每个动作的精确性
   - 类似于Visual-ERM的多层次反馈

### 4. Visual-ERM的局限性和未来方向

**局限性**:

虽然Visual-ERM是一个重要进步，但仍有一些局限：

1. **渲染器依赖**: 依赖高质量的渲染器，不适用所有任务
2. **计算开销**: 渲染和细粒度评估增加计算成本
3. **数据需求**: 需要大量细粒度标注数据
4. **任务范围**: 目前只覆盖Chart/Table/SVG

**未来方向**:

1. **扩展任务领域**:
   - UI设计: 将Figma/Sketch设计转换为代码
   - 3D场景: 从2D图像生成3D场景
   - 动画: 从视觉描述生成动画代码
   - 更广泛的vision-to-code任务

2. **改进效率**:
   - 增量渲染: 只重新渲染变化的部分
   - 稀疏评估: 只评估关键差异
   - 轻量化模型: 使用知识蒸馏等技术

3. **更强的泛化**:
   - 零样本奖励: 为未见过的任务提供奖励
   - 少样本适应: 快速适应新任务
   - 在线学习: 在使用中持续改进

4. **集成到更大系统**:
   - 端到端训练: 将Visual-ERM集成到端到端的训练流程
   - 多智能体: 评估多智能体协作的质量
   - 长期规划: 结合细粒度反馈和长期规划

**对Spatial AGI的启示**:

Visual-ERM的局限性和未来方向对Spatial AGI也有启示：

1. **任务扩展**: 将Visual-ERM的概念扩展到更多空间任务
2. **效率优化**: 提高空间评估和反馈的效率
3. **泛化能力**: 设计能够跨空间任务泛化的奖励模型
4. **系统集成**: 将细粒度空间反馈集成到完整的Spatial AGI系统

## 关键数据

### 1. 模型参数和规模

**Visual-ERM**:
- 基础模型: Qwen3-VL-8B-Instruct
- 参数量: 8B
- 训练数据规模: 340K实例 (104K Chart + 125K Table + 111K SVG)

**对比模型**:
- Qwen3-VL-235B-Instruct: 235B参数（比Visual-ERM大29倍）
- VinciCoder-8B-SFT: 8B参数
- JanusCoderV-8B: 8B参数

**关键观察**: 8B的Visual-ERM在细粒度差异判断上显著优于235B的通用模型，证明针对性训练比简单缩放更有效。

### 2. 数据集

**奖励模型训练数据**:
- Chart-to-Code: 104K实例
- Table-to-Markdown: 125K实例
- SVG-to-Code: 111K实例
- 总计: 340K实例

**RL训练数据**:
- Chart-to-Code: 11K图像
- Table-to-Markdown: 40K图像
- SVG-to-Code: 10K图像
- 总计: 61K图像

**VC-RewardBench**:
- 候选池: 4.5K图像对
- 最终实例: 1,335个高质量实例
- 分布: 595 Chart + 298 Table + 442 SVG

**关键观察**: RL训练数据远少于奖励模型训练数据，说明奖励模型训练需要更多细粒度标注，而RL训练只需要原始图像。

### 3. 训练设置

**Visual-ERM训练**:
- 学习率: 1×10^-5
- 批次大小: 32
- 训练轮数: 3 epochs
- 优化算法: 监督微调（SFT）
- 目标: 负对数似然（NLL）

**RL训练**:
- 学习率: 1×10^-6
- 批次大小: 256
- 采样: 每个GRPO更新步骤采样8个rollouts
- 优化算法: GRPO（Group Relative Policy Optimization）
- 正则化: KL散度（β控制强度）

**框架**:
- 训练实现: LLaMA-Factory
- RL实现: VERL

## 总结

### 核心发现总结

1. **细粒度视觉奖励是必要的**: 现有的文本规则和粗粒度视觉相似性都不足以支持vision-to-code RL

2. **可解释奖励的价值**: 细粒度、可解释的反馈不仅对人类有用，对机器学习和优化也更有价值

3. **跨任务泛化**: 通过多任务训练，Visual-ERM实现了跨任务泛化，降低了为每个任务单独设计奖励的成本

4. **训练和推理两用**: Visual-ERM不仅用于RL训练，还支持测试时扩展，在两个层面都带来性能提升

5. **专业化 > 缩放**: 针对性训练的8B模型在细粒度差异判断上显著优于简单缩放的235B模型

### 对Spatial AGI的意义

**直接贡献**:

Vision-to-code是Spatial AGI的一个重要组成部分，Visual-ERM显著提升了相关任务的质量：
- 图表解析: +8.4
- 表格解析: +2.7

这些提升直接转化为Spatial AGI在视觉理解和代码生成方面的能力提升。

**技术启发**:

Visual-ERM提供了多个对Spatial AGI有价值的技术启发：

1. **细粒度评估**: Spatial AGI需要细粒度的空间评估能力
2. **跨模态推理**: 结合视觉、语言、空间等多模态信息
3. **可解释反馈**: 提供类型、位置、严重程度等结构化反馈
4. **任务无关奖励**: 设计跨任务泛化的奖励模型

**应用场景**:

Visual-ERM可以应用于Spatial AGI的多个场景：

1. **机器人视觉执行**: 评估操作代码与场景视觉的保真度
2. **AR/VR内容生成**: 评估UI设计和3D场景的渲染质量
3. **智能文档处理**: 评估文档解析和图表理解的准确性
4. **智能制造和质检**: 评估产品与设计规格的视觉一致性

**未来方向**:

基于Visual-ERM的经验，Spatial AGI的未来发展方向包括：

- 例如: ding2025arm, li2025one, peng2025agentic

**Visual-ERM的贡献**:
- 大多数先前的RM专注于文本生成（写作、对话）
- 不支持视觉到代码任务，其中质量主要由视觉保真度决定
- Visual-ERM是专门为视觉到代码任务设计的视觉等价奖励模型

### Vision-to-Code任务

**Chart-to-Code**:
- 目标：将图表解析为能够忠实地重现原始图表的Python程序
- 代表工作: zhao2025chartcoder, tan2025chartmaster, zhang2025enhancing

**Table-to-Markdown**:
- 目标：将表格图像转换为结构化格式（如Markdown或HTML）
- 代表工作: ling2025table2latex_doc_rl, zhang2025monkeyocr_doc_rl, niu2025mineru2

**SVG-to-Code**:
- 目标：将矢量图形转换为代码表示
- 代表工作: li2025unisvg, yang2024omnisvg

**Visual-ERM的贡献**:
- 统一了多个vision-to-code任务的奖励建模
- 提供了任务无关的细粒度奖励信号

### RL for Vision-to-Code

**挑战**:
- 监督微调（SFT）依赖大规模、高质量数据集，成本高
- RL有前景，但现有奖励设计有严重缺陷

**现有奖励方法**:

1. **文本规则奖励**:
   - 在文本空间评估字符串级或结构代理
   - 不直接利用视觉证据
   - 可能引入模态偏差

2. **视觉编码器相似性奖励**:
   - 如DINO特征相似性
   - 比较视觉编码器提取的表示
   - 通常是粗粒度的，可解释性有限

**Visual-ERM的贡献**:
- 跨模态奖励模型
- 为Vision-to-Code提供细粒度、可解释、任务无关的反馈
